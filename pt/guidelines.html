Diretrizes internas de segurança da API

Listas de verificação de segurança, perspectivas de revisão de código e modelos de políticas para equipes de desenvolvimento.

📋 Lista de verificação de pré-desenvolvimento

Itens a serem verificados antes de desenvolver um novo endpoint de API.

  • A especificação da API (OpenAPI) foi criada
  • Os métodos de autenticação e autorização foram determinados
  • Tipos de parâmetros de entrada e restrições foram definidos
  • Os campos a serem incluídos nas respostas foram claramente definidos
  • O formato de resposta a erros é padronizado
  • Requisitos de limitação de taxa foram estabelecidos
  • Foi confirmado o tratamento de dados pessoais e sensíveis
  • O método de autenticação foi determinado (OAuth 2.0, JWT, chave API)
  • As permissões necessárias (escopos/funções) foram definidas para cada endpoint
  • As verificações de autorização em nível de recurso foram projetadas (contramedida BOLA)
  • As funções administrativas estão claramente separadas
  • Os métodos de expiração e atualização do token foram determinados
  • A comunicação usa TLS 1.2 ou superior
  • O método de criptografia para dados armazenados foi determinado
  • Foram determinadas políticas de mascaramento e anonimato de informações pessoais
  • Informações confidenciais (tokens, senhas) não são incluídas nos logs
  • As informações de depuração e os rastreamentos de pilha não estão incluídos nas respostas da API

📜 Modelos de Política de Segurança

Modelos para políticas básicas no desenvolvimento interno de APIs. Personalize para se adequar ao seu projeto.

1. Política de autenticação

PolicyModelo
■ Métodos de autenticação 
- APIs voltadas para o usuário: OAuth 2.0 (código de autorização + PKCE) 
- APIs servidor para servidor: OAuth 2.0 (credenciais de cliente) ou mTLS 
- APIs de integração externa: chave API + assinatura HMAC 

■ Gerenciamento de tokens 
- Expiração do token de acesso: 15 minutos 
- Expiração do token de atualização: 7 dias (rotação necessária) 
- Armazenamento de token: HttpOnly + Seguro + Cookie SameSite 

■ Política de senha 
- Mínimo de 12 caracteres, pelo menos uma maiúscula, minúscula, número e símbolo 
- Hash com bcrypt (fator de custo 12 ou superior) 
- Proteção contra ataques de lista de senhas (integração da API Have I Been Pwned)

2. Política de Design de API

PolicyModelo
■ Controle de versão 
- Gerenciamento de versões via caminho URL: /api/v1/resources 
- Período mínimo de suporte de 6 meses para versões antigas 
- Notificar a suspensão de uso por meio do cabeçalho de suspensão de uso 

■ Limitação de taxa (valores padrão) 
- APIs gerais: 100 solicitações/15 min 
- APIs de autenticação: 5 solicitações/15 min 
- APIs públicas: 30 solicitações/min 
- Sempre retorne cabeçalhos RateLimit-* 

■ Respostas 
- Tipo de conteúdo: application/json (fixo) 
- Respostas de erro no formato RFC 7807 (detalhes do problema) 
- Nunca retorne stack traces em produção 
- Paginação: padrão 20 itens, máximo de 100 itens

3. Política de registro e auditoria

PolicyModelo
■ Campos de registro obrigatórios 
- Carimbo de data e hora (ISO 8601, UTC) 
- ID da solicitação (UUID para rastreabilidade) 
- ID do usuário/ID do cliente API 
- Método HTTP + Endpoint 
- Código de status 
- IP de origem 
- Tempo de resposta 

■ Campos de registro proibidos (dados confidenciais) 
- Senhas/Tokens/Chaves API 
- Números de cartão de crédito 
- Exibição completa de PII (é necessário mascarar) 

■ Alertas de monitoramento 
- Falhas de autenticação: Alerta em 5+/min 
- Erros 403: Alerta a 10+/min 
- 500 erros: Alerta imediato em 1 ocorrência 
- Limite de taxa excedido: análise de padrões

🚨 Fluxo de resposta a incidentes

Fase 1: Detecção

Detecte incidentes por meio de alertas de monitoramento, relatórios de usuários ou notificações externas. Determine a gravidade por meio da triagem inicial.

0-30 minutos

Fase 2: Contenção

Evitar a escalada de danos. Invalide chaves de API, suspenda temporariamente os endpoints afetados e identifique o escopo do impacto.

30 minutos - 2 horas

Fase 3: Erradicação e Recuperação

Corrija vulnerabilidades, aplique patches e restaure serviços. Notifique os usuários afetados.

2-24 horas

Fase 4: Análise Pós-Incidente

Conduza post-mortem. Execute análises de causa raiz, desenvolva medidas de prevenção de recorrência e documente as descobertas.

1 a 5 dias úteis
