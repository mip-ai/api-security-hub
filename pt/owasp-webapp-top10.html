OWASP Web Application Top 10 (2025)

Os 10 riscos de segurança mais críticos para aplicações web e como mitigá-los.

O que é o OWASP Web Application Top 10?

O OWASP Top 10 é o documento de conscientização mais amplamente reconhecido sobre segurança de aplicações web. A edição de 2025 reflete o cenário de ameaças mais recente, introduzindo novas categorias como Falhas na Cadeia de Suprimentos de Software e Tratamento Inadequado de Condições Excepcionais.

1️⃣ A01 - Controle de Acesso Quebrado

Critical

Visão Geral

O controle de acesso aplica políticas para que os usuários não possam agir fora de suas permissões pretendidas. Falhas normalmente levam à divulgação não autorizada de informações, modificação ou destruição de dados, ou execução de funções fora dos limites do usuário.

Risco

Atacantes podem explorar falhas de controle de acesso para acessar contas de outros usuários, visualizar arquivos sensíveis, modificar dados de outros usuários ou alterar direitos de acesso.

Exemplo de Código Vulnerável

Python ❌ Bad 
# User ID taken directly from request without authorization check
@app.route('/api/user/<user_id>/profile')
def get_profile(user_id):
    user = db.get_user(user_id)  # No ownership check!
    return jsonify(user.to_dict())

Exemplo de Código Seguro

Python ✅ Good 
@app.route('/api/user/<user_id>/profile')
@login_required
def get_profile(user_id):
    # Verify the requesting user owns this resource
    if current_user.id != user_id and not current_user.is_admin:
        abort(403)

    user = db.get_user(user_id)
    if not user:
        abort(404)
    return jsonify(user.to_dict())

Lista de Verificação de Mitigação

  • Negar acesso por padrão, exceto para recursos públicos
  • Implementar verificações de controle de acesso do lado do servidor em cada requisição
  • Aplicar propriedade de registro em vez de aceitar IDs enviados pelo usuário
  • Registrar falhas de controle de acesso e alertar administradores sobre violações repetidas

2️⃣ A02 - Configuração Incorreta de Segurança

High

Visão Geral

A configuração incorreta de segurança ocorre quando as configurações de segurança são definidas, implementadas ou mantidas incorretamente. Isso inclui falta de hardening de segurança, recursos desnecessários habilitados, contas padrão com senhas inalteradas, mensagens de erro excessivamente detalhadas e cabeçalhos de segurança HTTP mal configurados.

Risco

Servidores, frameworks ou serviços em nuvem mal configurados podem expor dados sensíveis, permitir acesso não autorizado ou fornecer aos atacantes informações para planejar ataques adicionais.

Exemplo de Código Vulnerável

Python ❌ Bad 
# Debug mode enabled in production, verbose errors exposed
app = Flask(__name__)
app.config['DEBUG'] = True  # Exposes stack traces!
app.config['SECRET_KEY'] = 'default-secret'  # Default key!

Exemplo de Código Seguro

Python ✅ Good 
import os

app = Flask(__name__)
app.config['DEBUG'] = False
app.config['SECRET_KEY'] = os.environ['SECRET_KEY']

@app.after_request
def set_security_headers(response):
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['X-Frame-Options'] = 'DENY'
    response.headers['Strict-Transport-Security'] = 'max-age=31536000; includeSubDomains'
    response.headers['Content-Security-Policy'] = "default-src 'self'"
    return response

Lista de Verificação de Mitigação

  • Implementar um processo de hardening repetível para todos os ambientes
  • Remover ou desabilitar recursos, frameworks e contas padrão desnecessários
  • Configurar cabeçalhos de segurança adequados (CSP, HSTS, X-Frame-Options)
  • Usar ferramentas automatizadas para verificar a eficácia das configurações

3️⃣ A03 - Falhas na Cadeia de Suprimentos de Software

High

Visão Geral

Novo em 2025. Foca nos riscos relacionados a componentes de terceiros, dependências e pipelines CI/CD. Atacantes visam a cadeia de suprimentos de software comprometendo pacotes, injetando código malicioso em bibliotecas de código aberto ou explorando vulnerabilidades do pipeline de compilação.

Risco

Uma única dependência comprometida pode afetar milhares de aplicações. Ataques à cadeia de suprimentos podem levar a roubo de dados, instalação de backdoors ou comprometimento completo do sistema com detecção mínima.

Exemplo de Código Vulnerável

JavaScript ❌ Bad 
// package.json with unpinned dependencies
{
  "dependencies": {
    "express": "*",           // Any version!
    "lodash": "^4.0.0",       // Wide range
    "unknown-pkg": "^1.0.0"   // Unvetted package
  }
}

Exemplo de Código Seguro

JavaScript ✅ Good 
// package.json with pinned versions + lockfile + audit
{
  "dependencies": {
    "express": "4.21.2",
    "lodash": "4.17.21"
  },
  "scripts": {
    "preinstall": "npx npm-audit-resolver",
    "integrity-check": "npm audit signatures"
  }
}
// Also: use package-lock.json, enable Dependabot/Renovate,
// generate SBOM, verify package provenance

Lista de Verificação de Mitigação

  • Fixar versões de dependências e usar lock files para builds reproduzíveis
  • Habilitar verificação automatizada de dependências (Dependabot, Snyk, Renovate)
  • Gerar e manter uma Lista de Materiais de Software (SBOM)
  • Verificar a procedência e assinaturas dos pacotes antes da inclusão

4️⃣ A04 - Falhas Criptográficas

High

Visão Geral

Falhas relacionadas à criptografia que frequentemente levam à exposição de dados sensíveis. Isso inclui o uso de algoritmos obsoletos, geração fraca de chaves, falta de criptografia para dados em trânsito ou em repouso, e validação incorreta de certificados.

Risco

Criptografia fraca ou ausente pode expor senhas, números de cartão de crédito, registros médicos, informações pessoais e segredos comerciais, potencialmente levando a violações regulatórias (GDPR, PCI DSS).

Exemplo de Código Vulnerável

Python ❌ Bad 
import hashlib

# Storing passwords with weak hashing
def store_password(password: str) -> str:
    return hashlib.md5(password.encode()).hexdigest()  # MD5 is broken!

Exemplo de Código Seguro

Python ✅ Good 
import bcrypt

def store_password(password: str) -> bytes:
    # Use bcrypt with automatic salting
    salt = bcrypt.gensalt(rounds=12)
    return bcrypt.hashpw(password.encode(), salt)

def verify_password(password: str, hashed: bytes) -> bool:
    return bcrypt.checkpw(password.encode(), hashed)

Lista de Verificação de Mitigação

  • Usar algoritmos fortes e atuais (AES-256, bcrypt/argon2, SHA-256+)
  • Criptografar todos os dados sensíveis em trânsito (TLS 1.2+) e em repouso
  • Nunca usar algoritmos obsoletos (MD5, SHA-1, DES, RC4)
  • Implementar gerenciamento adequado de chaves com políticas de rotação

5️⃣ A05 - Injeção

Critical

Visão Geral

Falhas de injeção ocorrem quando dados não confiáveis são enviados a um interpretador como parte de um comando ou consulta. Injeção SQL, injeção NoSQL, injeção de comandos do SO e Cross-Site Scripting (XSS) são as formas mais comuns. Dados hostis podem enganar o interpretador para executar comandos não intencionais.

Risco

A injeção pode resultar em perda de dados, corrupção, acesso não autorizado, tomada completa do host ou negação de serviço. A injeção SQL sozinha continua sendo um dos vetores de ataque mais perigosos e prevalentes.

Exemplo de Código Vulnerável

Python ❌ Bad 
# SQL injection via string concatenation
def get_user(username: str):
    query = f"SELECT * FROM users WHERE name = '{username}'"
    return db.execute(query)  # username = "' OR '1'='1"

Exemplo de Código Seguro

Python ✅ Good 
# Parameterized queries prevent SQL injection
def get_user(username: str):
    query = "SELECT * FROM users WHERE name = %s"
    return db.execute(query, (username,))

# For XSS prevention, escape output
from markupsafe import escape

def render_comment(comment: str) -> str:
    return f"<p>{escape(comment)}</p>"

Lista de Verificação de Mitigação

  • Usar consultas parametrizadas ou declarações preparadas para todas as operações de banco de dados
  • Usar codificação de saída consciente do contexto para prevenir XSS
  • Validar e sanitizar todas as entradas do usuário no lado do servidor
  • Usar ORMs com consultas parametrizadas em vez de construção de strings SQL brutas

6️⃣ A06 - Design Inseguro

High

Visão Geral

Design inseguro refere-se a riscos relacionados a falhas de design e arquitetura. Requer o uso de modelagem de ameaças, padrões de design seguros e arquiteturas de referência. Um design inseguro não pode ser corrigido por uma implementação perfeita; os controles de segurança necessários nunca foram criados para se defender contra ataques específicos.

Risco

Falhas de design podem levar a vulnerabilidades de lógica de negócios difíceis de detectar com ferramentas automatizadas. Falta de limites de taxa em operações sensíveis, ausência de autenticação multifator para ações críticas ou detecção insuficiente de fraude são falhas em nível de design.

Exemplo de Código Vulnerável

Python ❌ Bad 
# Password reset with no rate limit or verification
@app.route('/reset-password', methods=['POST'])
def reset_password():
    email = request.form['email']
    new_pass = request.form['new_password']
    user = db.find_by_email(email)
    user.password = hash_password(new_pass)  # No token verification!
    db.save(user)

Exemplo de Código Seguro

Python ✅ Good 
from datetime import datetime, timedelta

@app.route('/reset-password', methods=['POST'])
@rate_limit("3/hour")
def reset_password():
    token = request.form['token']
    new_pass = request.form['new_password']

    # Verify time-limited, single-use token
    reset_req = db.find_reset_token(token)
    if not reset_req or reset_req.used or reset_req.expires < datetime.utcnow():
        abort(400, "Invalid or expired token")

    # Enforce password complexity
    if not meets_password_policy(new_pass):
        abort(400, "Password does not meet requirements")

    reset_req.user.password = hash_password(new_pass)
    reset_req.used = True
    db.save_all([reset_req.user, reset_req])

Lista de Verificação de Mitigação

  • Integrar modelagem de ameaças no ciclo de vida de desenvolvimento
  • Aplicar limitação de taxa e controles anti-automação em operações sensíveis
  • Usar padrões de design seguros e arquiteturas de referência
  • Escrever testes unitários e de integração para casos de abuso, não apenas casos de uso

7️⃣ A07 - Falhas de Autenticação

High

Visão Geral

A confirmação da identidade do usuário, autenticação e gerenciamento de sessão são críticos. Falhas de autenticação ocorrem quando aplicações permitem credential stuffing, força bruta, senhas fracas, ou possuem gerenciamento de sessão falho, como IDs de sessão que não rotacionam após o login.

Risco

Atacantes podem obter acesso a contas de usuários através de credential stuffing automatizado, força bruta ou sequestro de sessão. Contas comprometidas podem levar a roubo de identidade, fraude e violações de dados.

Exemplo de Código Vulnerável

Python ❌ Bad 
# No brute force protection, weak session handling
@app.route('/login', methods=['POST'])
def login():
    user = db.find_by_email(request.form['email'])
    if user and user.password == request.form['password']:  # Plain comparison!
        session['user'] = user.id  # Session ID not rotated
        return redirect('/dashboard')

Exemplo de Código Seguro

Python ✅ Good 
from flask_limiter import Limiter

limiter = Limiter(app, default_limits=["100/hour"])

@app.route('/login', methods=['POST'])
@limiter.limit("5/minute")
def login():
    user = db.find_by_email(request.form['email'])
    if not user or not bcrypt.checkpw(
        request.form['password'].encode(), user.password_hash
    ):
        # Generic error to prevent user enumeration
        return "Invalid credentials", 401

    # Rotate session ID after authentication
    session.regenerate()
    session['user'] = user.id

    # Check for MFA requirement
    if user.mfa_enabled:
        return redirect('/mfa-verify')
    return redirect('/dashboard')

Lista de Verificação de Mitigação

  • Implementar autenticação multifator (MFA) para todas as operações críticas
  • Aplicar proteção contra força bruta com limitação de taxa e bloqueio de conta
  • Rotacionar IDs de sessão após autenticação e usar flags de sessão seguras
  • Usar mensagens de erro genéricas para prevenir enumeração de usuários

8️⃣ A08 - Falhas de Integridade de Software ou Dados

High

Visão Geral

Falhas de integridade de software e dados se relacionam com código e infraestrutura que não protegem contra violações de integridade. Isso inclui deserialização insegura, uso de CDNs ou plugins não confiáveis sem verificação de integridade, e mecanismos de atualização automática sem atualizações assinadas.

Risco

A deserialização insegura pode levar à execução remota de código. Carregar scripts de CDNs não confiáveis sem Subresource Integrity (SRI) pode permitir que atacantes injetem código malicioso em sua aplicação.

Exemplo de Código Vulnerável

HTML ❌ Bad 
<!-- Loading external scripts without integrity checks -->
<script src="https://cdn.example.com/lib.js"></script>

<!-- Insecure deserialization in Python -->
import pickle
data = pickle.loads(user_input)  # Arbitrary code execution!

Exemplo de Código Seguro

HTML ✅ Good 
<!-- Use Subresource Integrity (SRI) for external scripts -->
<script src="https://cdn.example.com/lib.js"
  integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K..."
  crossorigin="anonymous"></script>

# Use safe deserialization in Python
import json
data = json.loads(user_input)  # Safe: only parses JSON data

Lista de Verificação de Mitigação

  • Usar Subresource Integrity (SRI) para todos os scripts e folhas de estilo externos
  • Nunca usar deserialização insegura (pickle, yaml.load) em entrada não confiável
  • Verificar assinaturas digitais em atualizações de software e artefatos CI/CD
  • Implementar verificações de integridade em pipelines CI/CD para prevenir adulteração

9️⃣ A09 - Falhas em Registro de Segurança e Alertas

Medium

Visão Geral

Sem registro, monitoramento e alertas suficientes, as violações não podem ser detectadas em tempo hábil. Registro insuficiente, integração ineficaz com sistemas de resposta a incidentes e falta de alertas em tempo real permitem que atacantes continuem atacando sistemas, mantenham persistência e adulterem ou extraiam dados.

Risco

Sem registro adequado, atacantes podem operar sem serem detectados por períodos prolongados. A maioria dos estudos de violação mostra que o tempo médio para detectar uma violação excede 200 dias, frequentemente descoberta por partes externas em vez de monitoramento interno.

Exemplo de Código Vulnerável

Python ❌ Bad 
# No logging of security-relevant events
@app.route('/login', methods=['POST'])
def login():
    user = authenticate(request.form)
    if not user:
        return "Login failed", 401  # No record of failure
    return redirect('/dashboard')

Exemplo de Código Seguro

Python ✅ Good 
import logging
from datetime import datetime

security_log = logging.getLogger('security')

@app.route('/login', methods=['POST'])
def login():
    user = authenticate(request.form)
    if not user:
        security_log.warning(
            "LOGIN_FAILED | ip=%s | email=%s | time=%s",
            request.remote_addr,
            request.form.get('email', 'unknown'),
            datetime.utcnow().isoformat(),
        )
        check_brute_force(request.remote_addr)
        return "Invalid credentials", 401

    security_log.info(
        "LOGIN_SUCCESS | user_id=%s | ip=%s",
        user.id, request.remote_addr,
    )
    return redirect('/dashboard')

Lista de Verificação de Mitigação

  • Registrar todas as tentativas de autenticação, falhas de controle de acesso e erros de validação de entrada
  • Usar um sistema centralizado de gerenciamento de logs com proteção contra adulteração
  • Implementar alertas em tempo real para padrões de atividade suspeita
  • Estabelecer um plano de resposta a incidentes com procedimentos de escalonamento definidos

🔟 A10 - Tratamento Inadequado de Condições Excepcionais

Medium

Visão Geral

Novo em 2025. Aplicações que tratam incorretamente erros, exceções e casos extremos podem expor informações sensíveis, entrar em estados inconsistentes ou criar condições exploráveis. Isso inclui mensagens de erro detalhadas em produção, exceções não capturadas que contornam controles de segurança e condições de corrida.

Risco

O tratamento inadequado de erros pode expor stack traces, detalhes de banco de dados ou caminhos internos. Condições de corrida em verificações de segurança podem permitir ataques TOCTOU (Time-of-Check-to-Time-of-Use), contornando autorização ou validação de pagamento.

Exemplo de Código Vulnerável

Python ❌ Bad 
# Exposing internal details in error messages
@app.route('/api/data')
def get_data():
    try:
        result = db.query(request.args['q'])
        return jsonify(result)
    except Exception as e:
        return jsonify({"error": str(e)}), 500  # Leaks DB details!

Exemplo de Código Seguro

Python ✅ Good 
import uuid, logging

logger = logging.getLogger(__name__)

@app.errorhandler(Exception)
def handle_exception(e):
    error_id = str(uuid.uuid4())
    logger.error("Unhandled exception [%s]: %s", error_id, e, exc_info=True)

    # Return generic error with reference ID for support
    return jsonify({
        "error": "An internal error occurred",
        "reference": error_id,
    }), 500

@app.route('/api/data')
def get_data():
    q = request.args.get('q')
    if not q or not is_valid_query(q):
        return jsonify({"error": "Invalid query parameter"}), 400

    result = db.query(q)
    return jsonify(result)

Lista de Verificação de Mitigação

  • Nunca expor stack traces, erros de banco de dados ou caminhos internos em produção
  • Implementar um manipulador de erros global que retorne mensagens genéricas com IDs de referência
  • Testar condições de corrida em operações críticas de segurança
  • Garantir que todos os caminhos de erro mantenham invariantes de segurança e limpem recursos
