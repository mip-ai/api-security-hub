⏱ Limitação de taxa

A limitação de taxa é um mecanismo de defesa fundamental para proteger seu sistema contra abuso de API, ataques DoS e scraping.

Implementação de limitação de taxa em Express.js

JavaScript (Express)Limitação de taxa multicamada
const rateLimit = require('express-rate-limit');
const RedisStore = require('rate-limit-redis');

// Limite global: aplicado a todas as APIs
const globalLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutos
  max: 100,
  standardHeaders: true,    // Retorna cabeçalhos RateLimit-*
  legacyHeaders: false,
  message: { error: 'Too many requests, please try again later.' },
});

// Para endpoints de autenticação: limites mais rígidos
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,  // Tentativas de login limitadas a 5 por 15 minutos
  skipSuccessfulRequests: true, //Solicitações bem-sucedidas não são contadas
});

// Para ambientes distribuídos: backend Redis
const distributedLimiter = rateLimit({
  store: new RedisStore({ sendCommand: (...args) => redisClient.sendCommand(args) }),
  windowMs: 60 * 1000,
  max: 30,
});

app.use(globalLimiter);
app.use('/api/auth', authLimiter);

Chaves de limitação de taxa

Ao aplicar limites de taxa usando chaves compostas, como chave de API, ID de usuário ou endpoint, além do endereço IP, você pode minimizar o impacto sobre usuários legítimos.

🔍 Validação de entrada

Todas as entradas da API podem conter dados maliciosos. Sempre valide no lado do servidor.

Princípios de Validação

Abordagem de lista de permissões

Defina uma lista de valores permitidos. Mais seguro que uma lista de bloqueios (lista de bloqueio). Pode lidar com novos padrões de ataque.

Tipo, comprimento e alcance

Verifique rigorosamente os tipos de dados, a contagem máxima de caracteres e os limites superior/inferior numéricos. Uma defesa fundamental contra buffer overflows.

Higienizar vs. Rejeitar

Rejeitar entradas inválidas é mais seguro do que "consertá-las". A higienização pode produzir transformações inesperadas.

Validação com esquema JSON

JavaScript (Express + Ajv)Validação de esquema
const Ajv = require('ajv');
const addFormats = require('ajv-formats');

const ajv = new Ajv({ allErrors: true, removeAdditional: true });
addFormats(ajv);

// Esquema para API de criação de usuário
const createUserSchema = {
  type: 'object',
  required: ['name', 'email'],
  additionalProperties: false,
  properties: {
    name: {
      type: 'string',
      minLength: 1,
      maxLength: 100,
      pattern: '^[a-zA-Z0-9\\s\\-]+$', //Restringe aos caracteres permitidos
    },
    email: {
      type: 'string',
      format: 'email',
      maxLength: 254,
    },
    age: {
      type: 'integer',
      minimum: 0,
      maximum: 150,
    },
  },
};

//Middleware de validação
function validateBody(schema) {
  const validate = ajv.compile(schema);
  return (req, res, next) => {
    if (!validate(req.body)) {
      return res.status(400).json({
        error: 'Validation failed',
        details: validate.errors,
      });
    }
    next();
  };
}

app.post('/api/users', validateBody(createUserSchema), createUser);

🌐 CORS (compartilhamento de recursos entre origens)

CORS é um mecanismo que controla a Política de Mesma Origem do navegador. A configuração incorreta pode levar a sérios riscos de segurança.

Configuração perigosa

Access-Control-Allow-Origin: * e Access-Control-Allow-Credentials: true não podem ser usados ​​juntos. Restrinja o uso de curingas apenas a APIs públicas.

Configuração segura do CORS

JavaScript (Express)Configuração do CORS
const cors = require('cors');

// Especifique explicitamente as origens permitidas
const allowedOrigins = [
  'https://app.example.com',
  'https://admin.example.com',
];

app.use(cors({
  origin(origin, callback) {
    //Permitir comunicação entre servidores (sem origem)
    if (!origin || allowedOrigins.includes(origin)) {
      callback(null, true);
    } else {
      callback(new Error('Not allowed by CORS'));
    }
  },
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization', 'X-API-Key'],
  credentials: true,
  maxAge: 86400,  // Cache de comprovação: 24 horas
}));

Lista de verificação do CORS

  • Não use um curinga (*) para Access-Control-Allow-Origin (para APIs que exigem autenticação)
  • Access-Control-Allow-Methods deve incluir apenas os métodos necessários
  • Access-Control-Allow-Headers deve incluir apenas os cabeçalhos necessários
  • Solicitações de simulação de cache com Access-Control-Max-Age
  • Restringir cabeçalhos de resposta expostos com Access-Control-Expose-Headers
  • Nunca reflita a origem da solicitação diretamente na resposta
