OWASP Web Application Top 10 (2025)

Los 10 riesgos de seguridad más críticos para aplicaciones web y cómo mitigarlos.

¿Qué es el OWASP Web Application Top 10?

El OWASP Top 10 es el documento de concienciación más reconocido sobre seguridad de aplicaciones web. La edición 2025 refleja el panorama de amenazas más reciente, introduciendo nuevas categorías como Fallos en la Cadena de Suministro de Software y Manejo Inadecuado de Condiciones Excepcionales.

1️⃣ A01 - Control de Acceso Roto

Critical

Descripción general

El control de acceso aplica políticas para que los usuarios no actúen fuera de sus permisos previstos. Las fallas típicamente conducen a la divulgación no autorizada de información, modificación o destrucción de datos, o la ejecución de funciones fuera de los límites del usuario.

Riesgo

Los atacantes pueden explotar fallas en el control de acceso para acceder a cuentas de otros usuarios, ver archivos sensibles, modificar datos de otros usuarios o cambiar derechos de acceso.

Ejemplo de Código Vulnerable

Python ❌ Bad 
# User ID taken directly from request without authorization check
@app.route('/api/user/<user_id>/profile')
def get_profile(user_id):
    user = db.get_user(user_id)  # No ownership check!
    return jsonify(user.to_dict())

Ejemplo de Código Seguro

Python ✅ Good 
@app.route('/api/user/<user_id>/profile')
@login_required
def get_profile(user_id):
    # Verify the requesting user owns this resource
    if current_user.id != user_id and not current_user.is_admin:
        abort(403)

    user = db.get_user(user_id)
    if not user:
        abort(404)
    return jsonify(user.to_dict())

Lista de Verificación de Mitigación

  • Denegar acceso por defecto excepto para recursos públicos
  • Implementar verificaciones de control de acceso del lado del servidor en cada solicitud
  • Aplicar la propiedad del registro en lugar de aceptar IDs enviados por el usuario
  • Registrar fallas de control de acceso y alertar a los administradores sobre violaciones repetidas

2️⃣ A02 - Configuración de Seguridad Incorrecta

High

Descripción general

La configuración incorrecta de seguridad ocurre cuando los ajustes de seguridad se definen, implementan o mantienen incorrectamente. Esto incluye falta de endurecimiento de seguridad, funciones innecesarias habilitadas, cuentas predeterminadas con contraseñas sin cambiar, mensajes de error demasiado detallados y encabezados de seguridad HTTP mal configurados.

Riesgo

Los servidores, frameworks o servicios en la nube mal configurados pueden exponer datos sensibles, permitir el acceso no autorizado o proporcionar a los atacantes información para planificar ataques adicionales.

Ejemplo de Código Vulnerable

Python ❌ Bad 
# Debug mode enabled in production, verbose errors exposed
app = Flask(__name__)
app.config['DEBUG'] = True  # Exposes stack traces!
app.config['SECRET_KEY'] = 'default-secret'  # Default key!

Ejemplo de Código Seguro

Python ✅ Good 
import os

app = Flask(__name__)
app.config['DEBUG'] = False
app.config['SECRET_KEY'] = os.environ['SECRET_KEY']

@app.after_request
def set_security_headers(response):
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['X-Frame-Options'] = 'DENY'
    response.headers['Strict-Transport-Security'] = 'max-age=31536000; includeSubDomains'
    response.headers['Content-Security-Policy'] = "default-src 'self'"
    return response

Lista de Verificación de Mitigación

  • Implementar un proceso de endurecimiento repetible para todos los entornos
  • Eliminar o deshabilitar funciones, frameworks y cuentas predeterminadas innecesarias
  • Configurar encabezados de seguridad adecuados (CSP, HSTS, X-Frame-Options)
  • Usar herramientas automatizadas para verificar la efectividad de las configuraciones

3️⃣ A03 - Fallos en la Cadena de Suministro de Software

High

Descripción general

Nuevo en 2025. Se centra en los riesgos relacionados con componentes de terceros, dependencias y pipelines CI/CD. Los atacantes apuntan a la cadena de suministro de software comprometiendo paquetes, inyectando código malicioso en bibliotecas de código abierto o explotando vulnerabilidades del pipeline de compilación.

Riesgo

Una sola dependencia comprometida puede afectar miles de aplicaciones. Los ataques a la cadena de suministro pueden llevar al robo de datos, instalación de puertas traseras o compromiso completo del sistema con detección mínima.

Ejemplo de Código Vulnerable

JavaScript ❌ Bad 
// package.json with unpinned dependencies
{
  "dependencies": {
    "express": "*",           // Any version!
    "lodash": "^4.0.0",       // Wide range
    "unknown-pkg": "^1.0.0"   // Unvetted package
  }
}

Ejemplo de Código Seguro

JavaScript ✅ Good 
// package.json with pinned versions + lockfile + audit
{
  "dependencies": {
    "express": "4.21.2",
    "lodash": "4.17.21"
  },
  "scripts": {
    "preinstall": "npx npm-audit-resolver",
    "integrity-check": "npm audit signatures"
  }
}
// Also: use package-lock.json, enable Dependabot/Renovate,
// generate SBOM, verify package provenance

Lista de Verificación de Mitigación

  • Fijar versiones de dependencias y usar archivos de bloqueo para compilaciones reproducibles
  • Habilitar escaneo automatizado de dependencias (Dependabot, Snyk, Renovate)
  • Generar y mantener una Lista de Materiales de Software (SBOM)
  • Verificar la procedencia y las firmas de los paquetes antes de su inclusión

4️⃣ A04 - Fallos Criptográficos

High

Descripción general

Fallos relacionados con la criptografía que a menudo conducen a la exposición de datos sensibles. Esto incluye el uso de algoritmos obsoletos, generación débil de claves, falta de cifrado para datos en tránsito o en reposo, y validación incorrecta de certificados.

Riesgo

La encriptación débil o faltante puede exponer contraseñas, números de tarjetas de crédito, registros médicos, información personal y secretos comerciales, potencialmente llevando a violaciones regulatorias (GDPR, PCI DSS).

Ejemplo de Código Vulnerable

Python ❌ Bad 
import hashlib

# Storing passwords with weak hashing
def store_password(password: str) -> str:
    return hashlib.md5(password.encode()).hexdigest()  # MD5 is broken!

Ejemplo de Código Seguro

Python ✅ Good 
import bcrypt

def store_password(password: str) -> bytes:
    # Use bcrypt with automatic salting
    salt = bcrypt.gensalt(rounds=12)
    return bcrypt.hashpw(password.encode(), salt)

def verify_password(password: str, hashed: bytes) -> bool:
    return bcrypt.checkpw(password.encode(), hashed)

Lista de Verificación de Mitigación

  • Usar algoritmos fuertes y actuales (AES-256, bcrypt/argon2, SHA-256+)
  • Cifrar todos los datos sensibles en tránsito (TLS 1.2+) y en reposo
  • Nunca usar algoritmos obsoletos (MD5, SHA-1, DES, RC4)
  • Implementar una gestión adecuada de claves con políticas de rotación

5️⃣ A05 - Inyección

Critical

Descripción general

Los fallos de inyección ocurren cuando datos no confiables se envían a un intérprete como parte de un comando o consulta. La inyección SQL, inyección NoSQL, inyección de comandos del SO y Cross-Site Scripting (XSS) son las formas más comunes. Los datos hostiles pueden engañar al intérprete para ejecutar comandos no deseados.

Riesgo

La inyección puede resultar en pérdida de datos, corrupción, acceso no autorizado, toma completa del host o denegación de servicio. La inyección SQL por sí sola sigue siendo uno de los vectores de ataque más peligrosos y prevalentes.

Ejemplo de Código Vulnerable

Python ❌ Bad 
# SQL injection via string concatenation
def get_user(username: str):
    query = f"SELECT * FROM users WHERE name = '{username}'"
    return db.execute(query)  # username = "' OR '1'='1"

Ejemplo de Código Seguro

Python ✅ Good 
# Parameterized queries prevent SQL injection
def get_user(username: str):
    query = "SELECT * FROM users WHERE name = %s"
    return db.execute(query, (username,))

# For XSS prevention, escape output
from markupsafe import escape

def render_comment(comment: str) -> str:
    return f"<p>{escape(comment)}</p>"

Lista de Verificación de Mitigación

  • Usar consultas parametrizadas o sentencias preparadas para todas las operaciones de base de datos
  • Usar codificación de salida consciente del contexto para prevenir XSS
  • Validar y sanear todas las entradas del usuario en el lado del servidor
  • Usar ORMs con consultas parametrizadas en lugar de construcción de cadenas SQL sin procesar

6️⃣ A06 - Diseño Inseguro

High

Descripción general

El diseño inseguro se refiere a riesgos relacionados con fallos de diseño y arquitectura. Requiere el uso de modelado de amenazas, patrones de diseño seguro y arquitecturas de referencia. Un diseño inseguro no puede ser corregido por una implementación perfecta; los controles de seguridad necesarios nunca se crearon para defenderse contra ataques específicos.

Riesgo

Los fallos de diseño pueden llevar a vulnerabilidades de lógica de negocio difíciles de detectar con herramientas automatizadas. La falta de límites de tasa en operaciones sensibles, la ausencia de autenticación multifactor para acciones críticas o la detección insuficiente de fraude son fallos a nivel de diseño.

Ejemplo de Código Vulnerable

Python ❌ Bad 
# Password reset with no rate limit or verification
@app.route('/reset-password', methods=['POST'])
def reset_password():
    email = request.form['email']
    new_pass = request.form['new_password']
    user = db.find_by_email(email)
    user.password = hash_password(new_pass)  # No token verification!
    db.save(user)

Ejemplo de Código Seguro

Python ✅ Good 
from datetime import datetime, timedelta

@app.route('/reset-password', methods=['POST'])
@rate_limit("3/hour")
def reset_password():
    token = request.form['token']
    new_pass = request.form['new_password']

    # Verify time-limited, single-use token
    reset_req = db.find_reset_token(token)
    if not reset_req or reset_req.used or reset_req.expires < datetime.utcnow():
        abort(400, "Invalid or expired token")

    # Enforce password complexity
    if not meets_password_policy(new_pass):
        abort(400, "Password does not meet requirements")

    reset_req.user.password = hash_password(new_pass)
    reset_req.used = True
    db.save_all([reset_req.user, reset_req])

Lista de Verificación de Mitigación

  • Integrar el modelado de amenazas en el ciclo de vida del desarrollo
  • Aplicar limitación de tasa y controles anti-automatización en operaciones sensibles
  • Usar patrones de diseño seguro y arquitecturas de referencia
  • Escribir pruebas unitarias y de integración para casos de abuso, no solo casos de uso

7️⃣ A07 - Fallos de Autenticación

High

Descripción general

La confirmación de la identidad del usuario, la autenticación y la gestión de sesiones son críticas. Los fallos de autenticación ocurren cuando las aplicaciones permiten credential stuffing, fuerza bruta, contraseñas débiles, o tienen gestión de sesiones defectuosa como IDs de sesión que no rotan después del inicio de sesión.

Riesgo

Los atacantes pueden acceder a cuentas de usuario mediante credential stuffing automatizado, fuerza bruta o secuestro de sesión. Las cuentas comprometidas pueden llevar a robo de identidad, fraude y violaciones de datos.

Ejemplo de Código Vulnerable

Python ❌ Bad 
# No brute force protection, weak session handling
@app.route('/login', methods=['POST'])
def login():
    user = db.find_by_email(request.form['email'])
    if user and user.password == request.form['password']:  # Plain comparison!
        session['user'] = user.id  # Session ID not rotated
        return redirect('/dashboard')

Ejemplo de Código Seguro

Python ✅ Good 
from flask_limiter import Limiter

limiter = Limiter(app, default_limits=["100/hour"])

@app.route('/login', methods=['POST'])
@limiter.limit("5/minute")
def login():
    user = db.find_by_email(request.form['email'])
    if not user or not bcrypt.checkpw(
        request.form['password'].encode(), user.password_hash
    ):
        # Generic error to prevent user enumeration
        return "Invalid credentials", 401

    # Rotate session ID after authentication
    session.regenerate()
    session['user'] = user.id

    # Check for MFA requirement
    if user.mfa_enabled:
        return redirect('/mfa-verify')
    return redirect('/dashboard')

Lista de Verificación de Mitigación

  • Implementar autenticación multifactor (MFA) para todas las operaciones críticas
  • Aplicar protección contra fuerza bruta con limitación de tasa y bloqueo de cuenta
  • Rotar IDs de sesión después de la autenticación y usar flags de sesión seguras
  • Usar mensajes de error genéricos para prevenir la enumeración de usuarios

8️⃣ A08 - Fallos de Integridad de Software o Datos

High

Descripción general

Los fallos de integridad de software y datos se relacionan con código e infraestructura que no protegen contra violaciones de integridad. Esto incluye deserialización insegura, uso de CDNs o plugins no confiables sin verificación de integridad, y mecanismos de actualización automática sin actualizaciones firmadas.

Riesgo

La deserialización insegura puede llevar a la ejecución remota de código. Cargar scripts de CDNs no confiables sin Subresource Integrity (SRI) puede permitir a los atacantes inyectar código malicioso en su aplicación.

Ejemplo de Código Vulnerable

HTML ❌ Bad 
<!-- Loading external scripts without integrity checks -->
<script src="https://cdn.example.com/lib.js"></script>

<!-- Insecure deserialization in Python -->
import pickle
data = pickle.loads(user_input)  # Arbitrary code execution!

Ejemplo de Código Seguro

HTML ✅ Good 
<!-- Use Subresource Integrity (SRI) for external scripts -->
<script src="https://cdn.example.com/lib.js"
  integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K..."
  crossorigin="anonymous"></script>

# Use safe deserialization in Python
import json
data = json.loads(user_input)  # Safe: only parses JSON data

Lista de Verificación de Mitigación

  • Usar Subresource Integrity (SRI) para todos los scripts y hojas de estilo externos
  • Nunca usar deserialización insegura (pickle, yaml.load) en entradas no confiables
  • Verificar firmas digitales en actualizaciones de software y artefactos CI/CD
  • Implementar verificaciones de integridad en pipelines CI/CD para prevenir manipulación

9️⃣ A09 - Fallos en Registro de Seguridad y Alertas

Medium

Descripción general

Sin suficiente registro, monitoreo y alertas, las brechas no pueden detectarse de manera oportuna. El registro insuficiente, la integración ineficaz con sistemas de respuesta a incidentes y la falta de alertas en tiempo real permiten a los atacantes continuar atacando sistemas, mantener la persistencia y manipular o extraer datos.

Riesgo

Sin un registro adecuado, los atacantes pueden operar sin ser detectados durante períodos prolongados. La mayoría de los estudios de brechas muestran que el tiempo promedio para detectar una brecha supera los 200 días, a menudo descubierta por partes externas en lugar de monitoreo interno.

Ejemplo de Código Vulnerable

Python ❌ Bad 
# No logging of security-relevant events
@app.route('/login', methods=['POST'])
def login():
    user = authenticate(request.form)
    if not user:
        return "Login failed", 401  # No record of failure
    return redirect('/dashboard')

Ejemplo de Código Seguro

Python ✅ Good 
import logging
from datetime import datetime

security_log = logging.getLogger('security')

@app.route('/login', methods=['POST'])
def login():
    user = authenticate(request.form)
    if not user:
        security_log.warning(
            "LOGIN_FAILED | ip=%s | email=%s | time=%s",
            request.remote_addr,
            request.form.get('email', 'unknown'),
            datetime.utcnow().isoformat(),
        )
        check_brute_force(request.remote_addr)
        return "Invalid credentials", 401

    security_log.info(
        "LOGIN_SUCCESS | user_id=%s | ip=%s",
        user.id, request.remote_addr,
    )
    return redirect('/dashboard')

Lista de Verificación de Mitigación

  • Registrar todos los intentos de autenticación, fallos de control de acceso y errores de validación de entrada
  • Usar un sistema centralizado de gestión de registros con protección contra manipulación
  • Implementar alertas en tiempo real para patrones de actividad sospechosa
  • Establecer un plan de respuesta a incidentes con procedimientos de escalamiento definidos

🔟 A10 - Manejo Inadecuado de Condiciones Excepcionales

Medium

Descripción general

Nuevo en 2025. Las aplicaciones que manejan incorrectamente errores, excepciones y casos límite pueden exponer información sensible, entrar en estados inconsistentes o crear condiciones explotables. Esto incluye mensajes de error detallados en producción, excepciones no capturadas que eluden controles de seguridad y condiciones de carrera.

Riesgo

El manejo inadecuado de errores puede exponer trazas de pila, detalles de base de datos o rutas internas. Las condiciones de carrera en verificaciones de seguridad pueden permitir ataques TOCTOU (Time-of-Check-to-Time-of-Use), eludiendo la autorización o validación de pagos.

Ejemplo de Código Vulnerable

Python ❌ Bad 
# Exposing internal details in error messages
@app.route('/api/data')
def get_data():
    try:
        result = db.query(request.args['q'])
        return jsonify(result)
    except Exception as e:
        return jsonify({"error": str(e)}), 500  # Leaks DB details!

Ejemplo de Código Seguro

Python ✅ Good 
import uuid, logging

logger = logging.getLogger(__name__)

@app.errorhandler(Exception)
def handle_exception(e):
    error_id = str(uuid.uuid4())
    logger.error("Unhandled exception [%s]: %s", error_id, e, exc_info=True)

    # Return generic error with reference ID for support
    return jsonify({
        "error": "An internal error occurred",
        "reference": error_id,
    }), 500

@app.route('/api/data')
def get_data():
    q = request.args.get('q')
    if not q or not is_valid_query(q):
        return jsonify({"error": "Invalid query parameter"}), 400

    result = db.query(q)
    return jsonify(result)

Lista de Verificación de Mitigación

  • Nunca exponer trazas de pila, errores de base de datos o rutas internas en producción
  • Implementar un manejador de errores global que devuelva mensajes genéricos con IDs de referencia
  • Probar condiciones de carrera en operaciones críticas de seguridad
  • Asegurar que todas las rutas de error mantengan las invariantes de seguridad y limpien los recursos
