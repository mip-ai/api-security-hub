Top 10 de seguridad de API de OWASP (2023)

Explica los 10 riesgos de seguridad más críticos específicos de las API y cómo mitigarlos.

¿Qué es el Top 10 de seguridad de API de OWASP?

Una clasificación de riesgos de seguridad específicos de API publicada por OWASP (Open Worldwide Application Security Project). La edición de 2023 proporciona la lista más reciente que refleja las amenazas que han aumentado con la adopción generalizada de las API.

1️⃣ API1:2023 - Autorización de nivel de objeto roto (BOLA)

Critical

Descripción general

Una vulnerabilidad que permite el acceso no autorizado a los datos de otros usuarios mediante la manipulación de ID de objetos. Se la conoce como la vulnerabilidad que ocurre con más frecuencia y es más fácil de explotar.

Riesgo

Los atacantes pueden acceder a la información personal de otros usuarios, detalles de pago, historial de pedidos y más simplemente cambiando el ID en la URL o los parámetros de solicitud.

Ejemplo de código vulnerable

JavaScript (Express) ❌ Bad 
// Recibe ID y devuelve datos directamente - sin verificación de autorización
app.get('/api/users/:id/orders', async (req, res) => {
  const orders = await Order.find({ userId: req.params.id });
  res.json(orders);
});

Ejemplo de código seguro

JavaScript (Express) ✅ Good 
// Verificar con el ID del usuario autenticado para verificar la autorización
app.get('/api/users/:id/orders', authenticate, async (req, res) => {
  if (req.user.id !== req.params.id && !req.user.isAdmin) {
    return res.status(403).json({ error: 'Forbidden' });
  }
  const orders = await Order.find({ userId: req.params.id });
  res.json(orders);
});

Lista de verificación de mitigación

  • Implementar controles de autorización a nivel de objeto en todos los puntos finales
  • Utilice UUID en lugar de ID secuenciales predecibles
  • Centralizar la lógica de autorización en middleware
  • Detecte omisiones de autorización con pruebas automatizadas

2️⃣ API2:2023: autenticación rota

Critical

Descripción general

Una vulnerabilidad que permite a los atacantes apoderarse de cuentas de usuarios legítimas debido a implementaciones defectuosas del mecanismo de autenticación.

Problemas comunes

Políticas de contraseñas débiles, gestión inadecuada de tokens, falta de protección de fuerza bruta, gestión de sesiones inadecuada y más.

JavaScript ✅ Ejemplo de mitigación: verificación JWT 
const jwt = require('jsonwebtoken');

function authenticate(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'Token required' });

  try {
    // Especifica explícitamente el algoritmo (evita el ataque alg: none)
    const decoded = jwt.verify(token, process.env.JWT_SECRET, {
      algorithms: ['HS256'],
      issuer: 'your-app',
    });
    req.user = decoded;
    next();
  } catch (err) {
    res.status(401).json({ error: 'Invalid token' });
  }
}
  • Especifique explícitamente el algoritmo al verificar las firmas JWT
  • Establezca tiempos de vencimiento de tokens cortos (tokens de acceso: dentro de 15 minutos)
  • Implementar la rotación de tokens de actualización
  • Aplicar limitación de la tasa de intentos de inicio de sesión

3️⃣ API3:2023: autorización de nivel de propiedad de objeto roto

High

Descripción general

Una vulnerabilidad donde el control de acceso a las propiedades (campos) de los objetos incluidos en las respuestas de la API es insuficiente. Este elemento combina exposición excesiva de datos y asignación masiva.

JavaScript ✅ Filtrado de respuestas 
// Devuelve sólo campos públicos del objeto de usuario
function sanitizeUser(user, requesterId) {
  const publicFields = { id: user.id, name: user.name, avatar: user.avatar };

  // Solo la propietaria puede ver campos adicionales.
  if (requesterId === user.id) {
    publicFields.email = user.email;
    publicFields.phone = user.phone;
  }

  return publicFields;
  // ❌ devolver usuario; filtraría contraseña_hash, etc.
}
  • Especifique explícitamente los campos para incluir en las respuestas de API (enfoque de lista de permitidos)
  • Restringir los campos del cuerpo de la solicitud aceptada mediante un esquema
  • Filtrar dinámicamente campos de respuesta según roles

4️⃣ API4:2023: consumo de recursos sin restricciones

High

Descripción general

Una vulnerabilidad en la que no hay límites en el tamaño, la frecuencia o el consumo de recursos de las solicitudes de API, lo que genera ataques DoS o aumentos de costos.

JavaScript (Express) ✅ Implementación de límites de recursos 
const rateLimit = require('express-rate-limit');

// Limitación de tasa global
app.use(rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutos
  max: 100,                   // Máximo 100 solicitudes
  standardHeaders: true,
}));

// límite de tamaño de carga útil
app.use(express.json({ limit: '10kb' }));

// límite superior de paginación
app.get('/api/items', (req, res) => {
  const limit = Math.min(parseInt(req.query.limit) || 20, 100);
  // ...
});

5️⃣ API5:2023: autorización de nivel de función rota

High

Descripción general

Una vulnerabilidad en la que no se implementa un control de acceso adecuado para las funciones administrativas y los puntos finales, lo que permite a los usuarios habituales realizar operaciones administrativas.

JavaScript ✅ Control de acceso basado en roles 
function authorize(...roles) {
  return (req, res, next) => {
    if (!roles.includes(req.user.role)) {
      return res.status(403).json({ error: 'Insufficient permissions' });
    }
    next();
  };
}

// Acceso solo de administrador
app.delete('/api/admin/users/:id', authenticate, authorize('admin'), deleteUser);

// Administradora y moderadora
app.put('/api/posts/:id/moderate', authenticate, authorize('admin', 'moderator'), moderatePost);

6️⃣ API6:2023: acceso sin restricciones a flujos comerciales confidenciales

Medium

Descripción general

Una vulnerabilidad en la que los flujos críticos de lógica empresarial no están protegidos contra ataques automatizados. Ejemplos: reventa de tickets, recolección masiva de cupones, publicación de spam, etc.

Mitigaciones

  • Implementar CAPTCHA/reCAPTCHA
  • Toma de huellas digitales del dispositivo
  • Tarifa a nivel empresarial que limita las operaciones
  • Detección de interacción humana

7️⃣ API7:2023: falsificación de solicitudes del lado del servidor (SSRF)

High

Descripción general

Una vulnerabilidad en la que el servidor obtiene una URL proporcionada por el usuario sin validación, lo que permite el acceso no autorizado a las redes internas.

JavaScript ✅ Mitigación de la SSRF 
const { URL } = require('url');

function isAllowedUrl(input) {
  try {
    const url = new URL(input);
    // Restringir protocolos
    if (!['https:', 'http:'].includes(url.protocol)) return false;
    // Bloquear rangos de IP privados
    const blocked = ['127.0.0.1', 'localhost', '0.0.0.0', '169.254.169.254'];
    if (blocked.includes(url.hostname)) return false;
    // Verifique los rangos de red interna (10.x, 172.16-31.x, 192.168.x)
    if (/^(10\.|172\.(1[6-9]|2\d|3[01])\.|192\.168\.)/.test(url.hostname)) return false;
    return true;
  } catch {
    return false;
  }
}

8️⃣ API8:2023: configuración incorrecta de seguridad

Medium

Descripción general

Configuraciones de seguridad inadecuadas en servidores o marcos API. Incluye permitir métodos HTTP innecesarios, políticas CORS demasiado permisivas, exposición de información de depuración y más.

JavaScript (Express) ✅ Configuración del encabezado de seguridad 
const helmet = require('helmet');
const cors = require('cors');

app.use(helmet());
app.use(cors({
  origin: ['https://app.example.com'], // No utilices comodines
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  credentials: true,
}));

// Manejador de errores: no devolver seguimientos de pila en producción
app.use((err, req, res, next) => {
  res.status(500).json({
    error: process.env.NODE_ENV === 'production'
      ? 'Internal Server Error'
      : err.message,
  });
});

9️⃣ API9:2023: gestión de inventario inadecuada

Medium

Descripción general

Un estado en el que las versiones antiguas de los terminales de API quedan expuestas sin que se apliquen parches de seguridad. También conocidas como API ocultas o API zombis.

Mitigaciones

  • Documentar y gestionar todos los puntos finales de API (OpenAPI/Swagger)
  • Establecer cronogramas claros de obsolescencia para API obsoletas
  • Bloquear puntos finales no registrados en la puerta de enlace API
  • Realizar auditorías periódicas del inventario de API

🔟 API10:2023: consumo inseguro de API

Medium

Descripción general

Una vulnerabilidad en la que se confía en las respuestas de API de terceros sin validación. Si una API externa se ve comprometida, el impacto se propaga a su propio sistema.

JavaScript ✅ Validación de respuesta de API externa 
const Ajv = require('ajv');
const ajv = new Ajv();

// Validar respuestas API externas con un esquema
const externalSchema = {
  type: 'object',
  required: ['id', 'status'],
  properties: {
    id: { type: 'string', maxLength: 50 },
    status: { type: 'string', enum: ['active', 'inactive'] },
  },
  additionalProperties: false,
};

const validate = ajv.compile(externalSchema);

async function fetchExternalData() {
  const res = await fetch('https://api.external.com/data');
  const data = await res.json();

  if (!validate(data)) {
    throw new Error('External API response validation failed');
  }
  return data;
}
