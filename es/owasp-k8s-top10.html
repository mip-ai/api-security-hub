1・鞘Ε K01 - Configuraciones Inseguras de Cargas de Trabajo

Critical

Descripciﾃｳn General

Las configuraciones inseguras de cargas de trabajo son el problema de seguridad mﾃ｡s comﾃｺn en Kubernetes. Riesgo

Un contenedor comprometido con privilegios de root y acceso al host puede escapar del sandbox del contenedor, acceder al sistema de archivos del host y pivotar a otras cargas de trabajo. La escalada de privilegios desde un pod mal configurado puede llevar a un compromiso total del clﾃｺster.

    securityContext:
      privileged: true            # Full host access!
      runAsUser: 0                # Running as root!
    volumeMounts:
    - mountPath: /host
      name: host-fs
  volumes:
  - name: host-fs
    hostPath:
      path: /                    # Entire host filesystem!

    runAsNonRoot: true
    seccompProfile:

        memory: "256Mi"

        memory: "128Mi"

Lista de Verificaciﾃｳn de Mitigaciﾃｳn

  • Ejecutar contenedores como no-root con readOnlyRootFilesystem y eliminar todas las capacidades
  • Establecer lﾃｭmites de recursos (CPU/memoria) para prevenir ataques de agotamiento de recursos
  • Usar digests de imagen inmutables en lugar de etiquetas mutables como :latest
  • Habilitar perfiles Seccomp y AppArmor/SELinux para defensa en profundidad

2・鞘Ε K02 - Configuraciones de Autorizaciﾃｳn Excesivamente Permisivas

Critical

Descripciﾃｳn General

El RBAC (Control de Acceso Basado en Roles) de Kubernetes es poderoso pero complejo. Los roles excesivamente permisivos 窶・especialmente los enlaces cluster-admin, permisos comodﾃｭn y privilegios excesivos de cuentas de servicio 窶・permiten el acceso no autorizado a recursos del clﾃｺster, secretos y cargas de trabajo.

Un atacante que obtiene acceso a una cuenta de servicio con privilegios excesivos puede listar secretos, crear pods privilegiados, modificar despliegues y escalar a cluster-admin. Las reglas RBAC con comodines son el equivalente en Kubernetes de otorgar acceso root.

  name: my-app            # App SA with cluster-admin!
  namespace: default

  kind: ClusterRole
  name: cluster-admin      # Full cluster control!
  apiGroup: rbac.authorization.k8s.io

Ejemplo de Cﾃｳdigo Seguro

YAML (Kubernetes RBAC) 笨・Good

  name: app-role
  namespace: my-app-ns

  resourceNames: ["app-config"]  # Named resource only

  name: app-role-binding
  namespace: my-app-ns

- kind: ServiceAccount
  name: my-app
  namespace: my-app-ns

  kind: Role
  name: app-role
  apiGroup: rbac.authorization.k8s.io

Lista de Verificaciﾃｳn de Mitigaciﾃｳn

  • Nunca enlazar cluster-admin a cuentas de servicio de aplicaciones; usar Roles con alcance de namespace
  • Evitar permisos comodﾃｭn (*) en reglas RBAC; especificar verbos y recursos exactos
  • Auditar regularmente enlaces RBAC con herramientas como kubectl-who-can o rbac-police
  • Deshabilitar el montaje automﾃ｡tico de tokens de cuenta de servicio cuando no sea necesario

3・鞘Ε K03 - Fallas en la Gestiﾃｳn de Secretos

Critical

Descripciﾃｳn General

Los Secrets de Kubernetes solo estﾃ｡n codificados en base64 por defecto, no encriptados. Lista de Verificaciﾃｳn de Mitigaciﾃｳn

  • Habilitar el cifrado en reposo para Kubernetes Secrets en etcd
  • Usar gestiﾃｳn externa de secretos (Vault, AWS Secrets Manager, External Secrets Operator)
  • Montar secretos como archivos en lugar de variables de entorno para reducir la exposiciﾃｳn
  • Restringir el acceso RBAC a secretos y auditar todo acceso a secretos con registro de auditorﾃｭa

4・鞘Ε K04 - Falta de Aplicaciﾃｳn de Polﾃｭticas a Nivel de Clﾃｺster

High

Descripciﾃｳn General

Sin polﾃｭticas a nivel de clﾃｺster, no hay barreras que impidan el despliegue de cargas de trabajo inseguras.   • Desplegar un motor de polﾃｭticas (Kyverno, OPA Gatekeeper) con modo enforce

  • Auditar violaciones de polﾃｭticas y revisar excepciones regularmente

5・鞘Ε K05 - Falta de Controles de Segmentaciﾃｳn de Red

High

Descripciﾃｳn General

Por defecto, todos los pods en un clﾃｺster de Kubernetes pueden comunicarse entre sﾃｭ sin restricciones. Sin segmentaciﾃｳn de red, el movimiento lateral es trivial. Un pod de frontend comprometido puede acceder directamente a bases de datos backend, servicios internos y la API de metadatos (169.254.169.254). Esto viola el principio de privilegio mﾃｭnimo en la capa de red.

Ejemplo de Cﾃｳdigo Vulnerable

  name: default-deny-all
  namespace: production

  podSelector: {}
  policyTypes: ["Ingress", "Egress"]

# Allow only frontend to backend communication

  name: allow-frontend-to-backend
  namespace: production

  podSelector:
    matchLabels:
      app: backend
  policyTypes: ["Ingress"]

        matchLabels:
          app: frontend

    - port: 8080

Lista de Verificaciﾃｳn de Mitigaciﾃｳn

  • Aplicar NetworkPolicies de denegaciﾃｳn por defecto tanto para ingress como egress en cada namespace
  • Permitir solo las rutas de comunicaciﾃｳn requeridas entre servicios
  • Bloquear el acceso de pods a las APIs de metadatos en la nube (169.254.169.254) mediante polﾃｭticas de red
  • Usar un plugin CNI que soporte la aplicaciﾃｳn de NetworkPolicy (Calico, Cilium)

6・鞘Ε K06 - Componentes de Kubernetes Excesivamente Expuestos

High

Descripciﾃｳn General

Los componentes del plano de control de Kubernetes (servidor API, etcd, kubelet, dashboard) y los servicios de aplicaciﾃｳn pueden quedar expuestos inadvertidamente a Internet o redes no confiables. Lista de Verificaciﾃｳn de Mitigaciﾃｳn

  • Nunca exponer el Dashboard de Kubernetes, el servidor API o etcd a Internet pﾃｺblica
  • Usar servicios ClusterIP y kubectl proxy/port-forward para acceso a componentes internos
  • Habilitar autenticaciﾃｳn en todos los componentes de Kubernetes (kubelet, servidor API, etcd)
  • Restringir el acceso al servidor API a redes confiables usando reglas de firewall o redes autorizadas

7・鞘Ε K07 - Componentes del Clﾃｺster Mal Configurados y Vulnerables

High

Descripciﾃｳn General

Los clﾃｺsteres de Kubernetes constan de muchos componentes: servidor API, etcd, kubelet, kube-proxy, CoreDNS y complementos de terceros (controladores Ingress, service meshes, monitoreo). 8・鞘Ε K08 - Movimiento Lateral del Clﾃｺster a la Nube

Critical

Descripciﾃｳn General

Los clﾃｺsteres de Kubernetes que se ejecutan en entornos de nube (AWS, GCP, Azure) pueden acceder a las APIs de metadatos de la nube y roles IAM adjuntos a los nodos. Un pod comprometido puede consultar la API de metadatos (169.254.169.254) para obtener credenciales de la nube y luego acceder a cualquier recurso en la nube que el rol del nodo permita. # Only this specific pod gets these specific permissions

Lista de Verificaciﾃｳn de Mitigaciﾃｳn

  • Usar IAM a nivel de pod (IRSA en AWS, Workload Identity en GCP/Azure) en lugar de roles a nivel de nodo
  • Bloquear el acceso de pods a la API de metadatos de la nube (169.254.169.254) mediante NetworkPolicy
  • Aplicar IMDSv2 (hop limit = 1) en AWS para evitar que los pods accedan a los metadatos del nodo
  • Aplicar polﾃｭticas IAM de privilegio mﾃｭnimo con alcance a recursos y acciones especﾃｭficos

9・鞘Ε K09 - Mecanismos de Autenticaciﾃｳn Rotos

Critical

Descripciﾃｳn General

Kubernetes soporta mﾃｺltiples mecanismos de autenticaciﾃｳn: certificados, tokens, OIDC y autenticaciﾃｳn webhook. Ejemplo de Cﾃｳdigo Vulnerable

Lista de Verificaciﾃｳn de Mitigaciﾃｳn

  • Usar integraciﾃｳn OIDC con su proveedor de identidad en lugar de tokens estﾃ｡ticos o certificados
  • Deshabilitar automountServiceAccountToken para pods que no necesitan acceso a la API de Kubernetes
  • Usar tokens de cuenta de servicio vinculados con tiempos de expiraciﾃｳn cortos
  • Rotar certificados y tokens regularmente; revocar acceso inmediatamente para usuarios dados de baja
