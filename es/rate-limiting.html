⏱ Limitación de velocidad

La limitación de velocidad es un mecanismo de defensa fundamental para proteger su sistema del abuso de API, ataques DoS y scraping.

Comparación de algoritmos

AlgoritmoCaracterísticasVentajasContras
Fixed WindowCuenta dentro de una ventana de tiempo fijaSimple de implementar, memoria eficienteLas explosiones ocurren en los límites de las ventanas.
Sliding Window LogRegistros de solicitudes de marcas de tiempoControl precisoAlto consumo de memoria
Sliding Window CounterPromedio ponderado de ventanas anteriores y actualesBuen equilibrioAlgo compleja
Token BucketConsume tokens para accederPermite ráfagas mientras se aplican límitesRequiere ajuste de parámetros
Leaky BucketProcesa solicitudes a un ritmo constanteTasa de salida estableMal manejando ráfagas

Implementación de limitación de velocidad en Express.js

JavaScript (Express)Limitación de velocidad multicapa
const rateLimit = require('express-rate-limit');
const RedisStore = require('rate-limit-redis');

// Límite global: aplicado a todas las API
const globalLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutos
  max: 100,
  standardHeaders: true,    // Devuelve encabezados RateLimit-*
  legacyHeaders: false,
  message: { error: 'Too many requests, please try again later.' },
});

// Para puntos finales de autenticación: límites más estrictos
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,  // Intentos de inicio de sesión limitados a 5 cada 15 minutos
  skipSuccessfulRequests: true, // Las solicitudes exitosas no se cuentan
});

// Para entornos distribuidos: backend de Redis
const distributedLimiter = rateLimit({
  store: new RedisStore({ sendCommand: (...args) => redisClient.sendCommand(args) }),
  windowMs: 60 * 1000,
  max: 30,
});

app.use(globalLimiter);
app.use('/api/auth', authLimiter);

Teclas de limitación de velocidad

Al aplicar límites de velocidad utilizando claves compuestas como clave API, ID de usuario o punto final además de la dirección IP, puede minimizar el impacto en los usuarios legítimos.

🔍 Validación de entrada

Todas las entradas de API pueden contener datos maliciosos. Valide siempre en el lado del servidor.

Principios de validación

Enfoque de lista blanca

Defina una lista de valores permitidos. Más seguro que una lista de denegación (lista de bloqueo). Puede manejar nuevos patrones de ataque.

Tipo, longitud y rango

Verifique estrictamente los tipos de datos, el número máximo de caracteres y los límites numéricos superior/inferior. Una defensa fundamental contra los desbordamientos del buffer.

Desinfectar versus rechazar

Rechazar una entrada no válida es más seguro que "arreglarla". La higienización puede producir transformaciones inesperadas.

Validación con esquema JSON

JavaScript (Express + Ajv)Validación de esquema
const Ajv = require('ajv');
const addFormats = require('ajv-formats');

const ajv = new Ajv({ allErrors: true, removeAdditional: true });
addFormats(ajv);

// Esquema para la API de creación de usuarios
const createUserSchema = {
  type: 'object',
  required: ['name', 'email'],
  additionalProperties: false,
  properties: {
    name: {
      type: 'string',
      minLength: 1,
      maxLength: 100,
      pattern: '^[a-zA-Z0-9\\s\\-]+$', // Restringir a caracteres permitidos
    },
    email: {
      type: 'string',
      format: 'email',
      maxLength: 254,
    },
    age: {
      type: 'integer',
      minimum: 0,
      maximum: 150,
    },
  },
};

// middleware de validación
function validateBody(schema) {
  const validate = ajv.compile(schema);
  return (req, res, next) => {
    if (!validate(req.body)) {
      return res.status(400).json({
        error: 'Validation failed',
        details: validate.errors,
      });
    }
    next();
  };
}

app.post('/api/users', validateBody(createUserSchema), createUser);

🌐 CORS (intercambio de recursos entre orígenes)

CORS es un mecanismo que controla la política del mismo origen del navegador. Una mala configuración puede provocar graves riesgos de seguridad.

Configuración peligrosa

Access-Control-Allow-Origin: * y Access-Control-Allow-Credentials: true no se pueden utilizar juntos. Restrinja el uso de comodines únicamente a las API públicas.

Configuración CORS segura

JavaScript (Express)Configuración CORS
const cors = require('cors');

// Especifica explícitamente los orígenes permitidos
const allowedOrigins = [
  'https://app.example.com',
  'https://admin.example.com',
];

app.use(cors({
  origin(origin, callback) {
    // Permitir comunicación de servidor a servidor (sin origen)
    if (!origin || allowedOrigins.includes(origin)) {
      callback(null, true);
    } else {
      callback(new Error('Not allowed by CORS'));
    }
  },
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization', 'X-API-Key'],
  credentials: true,
  maxAge: 86400,  // Caché de verificación previa: 24 horas
}));

Lista de verificación CORS

  • No utilice un comodín (*) para Access-Control-Allow-Origin (para API que requieren autenticación)
  • Access-Control-Allow-Methods debe incluir solo los métodos requeridos
  • Access-Control-Allow-Headers debe incluir solo los encabezados requeridos
  • Solicitudes de verificación previa en caché con Access-Control-Max-Age
  • Restrinja los encabezados de respuesta expuestos con Access-Control-Expose-Headers
  • Nunca refleje el origen de la solicitud directamente en la respuesta.

Referencia de encabezados CORS
