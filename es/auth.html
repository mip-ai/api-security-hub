Mejores prácticas de autenticación y autorización

Una guía práctica para la autenticación y autorización de API, que incluye OAuth 2.0, JWT y administración de claves de API.

🔑 OAuth 2.0

OAuth 2.0 es el marco de autorización más utilizado para el acceso a API. Seleccionar el tipo de subvención adecuado es fundamental.

Elegir un tipo de subvención

Tipo de subvención Caso de uso Nivel de seguridad
Authorization Code + PKCE SPA y aplicaciones móviles (recomendado) más alto
Authorization Code Aplicaciones web del lado del servidor Alta
Client Credentials Comunicación de servidor a servidor (M2M) Medio
Implícito (en desuso) SPA heredados Bajo
Contraseña del propietario del recurso (en desuso) Solo de primera mano altamente confiable Bajo
No utilice concesión implícita

La concesión implícita ahora está en desuso porque el token está expuesto en el fragmento de URL. Utilice el Código de autorización + PKCE para SPA.

Código de autorización + Flujo PKCE

JavaScript Generación de Retos PKCE 
// 1. Generar aleatoriamente code_verifier
function generateCodeVerifier() {
  const array = new Uint8Array(32);
  crypto.getRandomValues(array);
  return btoa(String.fromCharCode(...array))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}

// 2. Generar code_challenge con SHA-256
async function generateCodeChallenge(verifier) {
  const encoder = new TextEncoder();
  const data = encoder.encode(verifier);
  const digest = await crypto.subtle.digest('SHA-256', data);
  return btoa(String.fromCharCode(...new Uint8Array(digest)))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}

// 3. Solicitud de autorización
const verifier = generateCodeVerifier();
const challenge = await generateCodeChallenge(verifier);

const authUrl = `https://auth.example.com/authorize?` +
  `response_type=code&` +
  `client_id=${clientId}&` +
  `redirect_uri=${redirectUri}&` +
  `code_challenge=${challenge}&` +
  `code_challenge_method=S256&` +
  `scope=read write`;

🎫 JWT (token web JSON)

JWT se utiliza ampliamente como token de autenticación sin estado, pero los errores de implementación conducen directamente a vulnerabilidades de seguridad.

Consideraciones de seguridad de JWT

✅ Hacer

  • Especifique explícitamente el algoritmo (HS256 / RS256)
  • Establezca un tiempo de vencimiento (exp) corto (15 minutos o menos)
  • Validar emisor (iss) y audiencia (aud)
  • Generar claves secretas con longitud suficiente (256 bits o más)
  • Implementar rotación para tokens de actualización

❌ No lo hagas

  • Almacene información confidencial en la carga útil JWT (Base64 no es cifrado)
  • Permitir alg: "ninguna"
  • Almacenar tokens en localStorage (vulnerable a XSS)
  • Descuidar la gestión de revocación de tokens
  • Utilice una clave pública RS256 como clave secreta HS256

Implementación segura de JWT

JavaScript (Node.js) Emisión y verificación de tokens 
const jwt = require('jsonwebtoken');

// --- Emisión de tokens ---
function issueTokens(user) {
  const accessToken = jwt.sign(
    { sub: user.id, role: user.role },
    process.env.JWT_SECRET,
    {
      algorithm: 'HS256',
      expiresIn: '15m',
      issuer: 'api.example.com',
      audience: 'app.example.com',
    }
  );

  const refreshToken = jwt.sign(
    { sub: user.id, type: 'refresh' },
    process.env.REFRESH_SECRET,
    { algorithm: 'HS256', expiresIn: '7d' }
  );

  return { accessToken, refreshToken };
}

// --- Verificación de token ---
function verifyAccessToken(token) {
  return jwt.verify(token, process.env.JWT_SECRET, {
    algorithms: ['HS256'],   // ← ¡Especifica siempre!
    issuer: 'api.example.com',
    audience: 'app.example.com',
  });
}

Comparación de ubicaciones de almacenamiento de tokens

Ubicación de almacenamiento Resistencia XSS Resistencia CSRF Recomendación
HttpOnly Cookie ◎ △ (requiere configuración SameSite) Recomendada
Memoria (variable) ◎ ◎ Perdido en la navegación de la página
localStorage ✕ (se puede robar mediante XSS) ◎ No recomendado
sessionStorage ✕ (se puede robar mediante XSS) ◎ Uso limitado
Recomendada: Cookie HttpOnly + Secure + SameSite

Almacenar el token de acceso en una cookie con los atributos HttpOnly, Secure y SameSite=Strict es el método más seguro. Esto evita el robo de tokens mediante ataques XSS.

🗝 Gestión de claves API

Las claves API son un mecanismo de autenticación simple, pero una gestión inadecuada puede generar riesgos graves.

Mejores prácticas clave de API

Generación y Almacenamiento

Generar con suficiente entropía (256 bits o más). Almacenar hash; nunca mantenga texto sin formato en la base de datos.

Restricción de alcance

Asigne alcances mínimos por clave (solo lectura, solo recursos específicos, etc.). Evite los permisos comodín.

Rotación

Rote las claves periódicamente (dentro de los 90 días). Establezca un período de gracia para las llaves antiguas. Revocar inmediatamente en caso de fuga.

Método de transmisión

Enviar mediante encabezado (X-API-Key o Autorización). No incluir en los parámetros de consulta de URL (aparecen en los registros).

Gestión de variables de entorno

No codifique claves en el código fuente. Gestionarlos con variables de entorno o un administrador secreto.

Monitoreo y registro

Supervisar el uso de la clave API. Detectar patrones anómalos (solicitudes masivas, accesos desde IP desconocidas).

Ejemplo de implementación de validación de clave API

JavaScript (Express) Validación de clave API hash 
const crypto = require('crypto');

// Hash de la clave API con SHA-256 y comparar
async function validateApiKey(req, res, next) {
  const apiKey = req.headers['x-api-key'];
  if (!apiKey) {
    return res.status(401).json({ error: 'API key required' });
  }

  // Hash y búsqueda en la base de datos (timingSafeEqual no es necesario ya que comparamos hashes)
  const hashedKey = crypto
    .createHash('sha256')
    .update(apiKey)
    .digest('hex');

  const keyRecord = await db.findApiKey(hashedKey);

  if (!keyRecord || keyRecord.revokedAt) {
    return res.status(403).json({ error: 'Invalid API key' });
  }

  // Comprobar vencimiento
  if (keyRecord.expiresAt && new Date() > keyRecord.expiresAt) {
    return res.status(403).json({ error: 'API key expired' });
  }

  req.apiClient = keyRecord;
  next();
}

Método Apátrida Caso de uso Consideraciones de seguridad
OAuth 2.0 + PKCE ○ API que requieren autorización del usuario estado/PKCE requerido, gestión de vencimiento de token
JWT Bearer ○ Comunicación entre microservicios Se requiere especificación del algoritmo, no hay datos confidenciales en la carga útil
Clave API ○ Integraciones externas de servidor a servidor Se requiere restricción de rotación y alcance
mTLS ○ Comunicación M2M de alta seguridad Alto costo operativo para la gestión de certificados.
Cookie de sesión ✕ Aplicaciones web tradicionales Se requiere protección CSRF, desafíos de escalabilidad
