Pautas de seguridad de API internas

Listas de verificación de seguridad, perspectivas de revisión de código y plantillas de políticas para equipos de desarrollo.

📋 Lista de verificación previa al desarrollo

Elementos que se deben verificar antes de desarrollar un nuevo punto final de API.

  • Se ha creado la especificación API (OpenAPI)
  • Se han determinado los métodos de autenticación y autorización.
  • Se han definido los tipos de parámetros de entrada y las restricciones.
  • Los campos a incluir en las respuestas se han definido claramente.
  • El formato de respuesta de error está estandarizado
  • Se han establecido requisitos de limitación de tarifas.
  • Se ha confirmado el tratamiento de datos personales y sensibles.
  • Se ha determinado el método de autenticación (OAuth 2.0, JWT, clave API)
  • Se han definido los permisos necesarios (ámbitos/roles) para cada punto final
  • Se han diseñado controles de autorización a nivel de recursos (contramedida BOLA)
  • Las funciones administrativas están claramente separadas
  • Se han determinado los métodos de actualización y vencimiento del token.
  • La comunicación utiliza TLS 1.2 o superior
  • Se ha determinado el método de cifrado para los datos almacenados.
  • Se han determinado políticas de enmascaramiento y anonimización de información personal.
  • La información confidencial (tokens, contraseñas) no se incluye en los registros
  • La información de depuración y los seguimientos de la pila no se incluyen en las respuestas de la API.

📜 Plantillas de políticas de seguridad

Plantillas para políticas básicas en el desarrollo interno de API. Personalízalo para adaptarlo a tu proyecto.

1. Política de autenticación

PolicyPlantilla
■ Métodos de autenticación 
- API orientadas al usuario: OAuth 2.0 (Código de autorización + PKCE) 
- API de servidor a servidor: OAuth 2.0 (credenciales de cliente) o mTLS 
- API de integración externa: clave API + firma HMAC 

■ Gestión de tokens 
- Caducidad del token de acceso: 15 minutos 
- Caducidad del token de actualización: 7 días (se requiere rotación) 
- Almacenamiento de tokens: HttpOnly + Secure + SameSite Cookie 

■ Política de contraseñas 
- Mínimo 12 caracteres, al menos una mayúscula, una minúscula, un número y un símbolo 
- Hash con bcrypt (factor de costo 12 o superior) 
- Protección contra ataques de lista de contraseñas (integración de API Have I Been Pwned)

2. Política de diseño de API

PolicyPlantilla
■ Control de versiones 
- Gestión de versiones a través de la ruta URL: /api/v1/resources 
- Período mínimo de soporte de 6 meses para versiones antiguas 
- Notificar la obsolescencia a través del encabezado de obsolescencia 

■ Limitación de velocidad (valores predeterminados) 
- API generales: 100 solicitudes/15 min 
- API de autenticación: 5 solicitudes/15 min 
- API públicas: 30 solicitudes/min 
- Siempre devuelve encabezados RateLimit-* 

■ Respuestas 
- Tipo de contenido: aplicación/json (fijo) 
- Respuestas de error en formato RFC 7807 (Detalles del problema) 
- Nunca devolver rastros de pila en producción 
- Paginación: predeterminado 20 elementos, máximo 100 elementos

3. Política de registro y auditoría

PolicyPlantilla
■ Campos de registro obligatorios 
- Marca de tiempo (ISO 8601, UTC) 
- ID de solicitud (UUID para trazabilidad) 
- ID de usuario/ID de cliente API 
- Método HTTP + Punto final 
- Código de estado 
- IP de origen 
- Tiempo de respuesta 

■ Campos de registro prohibidos (datos confidenciales) 
- Contraseñas / Tokens / Claves API 
- Números de tarjetas de crédito 
- Visualización de PII completa (se requiere enmascaramiento) 

■ Alertas de seguimiento 
- Fallos de autenticación: alerta a 5+/min 
- Errores 403: Alerta a 10+/min 
- 500 errores: alerta inmediata en 1 ocurrencia 
- Límite de tasa excedido: análisis de patrones

🚨 Flujo de respuesta a incidentes

Fase 1: Detección

Detecta incidencias mediante alertas de seguimiento, informes de usuarios o notificaciones externas. Determinar la gravedad mediante la clasificación inicial.

0-30 minutos

Fase 2: Contención

Prevenir la escalada de daños. Invalide las claves API, suspenda temporalmente los puntos finales afectados e identifique el alcance del impacto.

30 minutos - 2 horas

Fase 3: Erradicación y Recuperación

Corrija vulnerabilidades, aplique parches y restaure servicios. Notificar a los usuarios afectados.

2-24 horas

Fase 4: Análisis posterior al incidente

Realizar autopsia. Realice análisis de causa raíz, desarrolle medidas de prevención de recurrencia y documente los hallazgos.

1-5 días hábiles
