OWASP Kubernetes Top 10

Kubernetes環境における最も重要な10のセキュリティリスクとその対策方法

OWASP Kubernetes Top 10とは

OWASP Kubernetes Top 10は、Kubernetes環境における最も重大なセキュリティリスクを特定しています。Kubernetesはコンテナオーケストレーションの事実上の標準となっていますが、その柔軟性と複雑性は数多くのセキュリティ課題をもたらします。本ガイドでは、ワークロードの設定ミス、RBACの問題、シークレット管理、ネットワーク分離などを網羅します。

1️⃣ K01 - 安全でないワークロード設定

Critical

概要

安全でないワークロード設定は、Kubernetesセキュリティにおける最も一般的な問題です。rootとして実行されるコンテナ、過剰な権限、書き込み可能なファイルシステム、リソース制限なしの設定は重大な攻撃面を生み出します。デフォルト設定は多くの場合安全ではなく、明示的に強化する必要があります。

リスク

root権限とホストアクセスを持つ侵害されたコンテナは、コンテナサンドボックスから脱出し、ホストファイルシステムにアクセスし、他のワークロードに侵入できます。設定ミスのあるPodからの権限昇格は、クラスタ全体の侵害につながる可能性があります。

脆弱なコード例

YAML (Kubernetes Pod) ❌ Bad 
apiVersion: v1
kind: Pod
metadata:
  name: insecure-app
spec:
  containers:
  - name: app
    image: myapp:latest          # Mutable tag!
    securityContext:
      privileged: true            # Full host access!
      runAsUser: 0                # Running as root!
    volumeMounts:
    - mountPath: /host
      name: host-fs
  volumes:
  - name: host-fs
    hostPath:
      path: /                    # Entire host filesystem!

安全なコード例

YAML (Kubernetes Pod) ✅ Good 
apiVersion: v1
kind: Pod
metadata:
  name: secure-app
spec:
  securityContext:
    runAsNonRoot: true
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: myapp@sha256:abc123...  # Immutable digest
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      runAsUser: 1000
      capabilities:
        drop: ["ALL"]
    resources:
      limits:
        cpu: "500m"
        memory: "256Mi"
      requests:
        cpu: "100m"
        memory: "128Mi"

対策チェックリスト

  • コンテナを非rootで実行し、readOnlyRootFilesystemを有効にし、すべてのケーパビリティを削除する
  • リソース制限(CPU/メモリ)を設定してリソース枯渇攻撃を防ぐ
  • :latestのような可変タグではなく、イミュータブルなイメージダイジェストを使用する
  • SeccompとAppArmor/SELinuxプロファイルを有効にして多層防御を実現する

2️⃣ K02 - 過度に寛容な認可設定

Critical

概要

KubernetesのRBAC(ロールベースアクセス制御)は強力ですが複雑です。過度に寛容なロール、特にcluster-adminバインディング、ワイルドカード権限、過剰なサービスアカウント権限は、クラスタリソース、シークレット、ワークロードへの不正アクセスを許可します。

リスク

過度な権限を持つサービスアカウントへのアクセスを得た攻撃者は、シークレットの一覧表示、特権付きPodの作成、デプロイメントの変更、cluster-adminへの昇格が可能です。ワイルドカードRBACルールは、Kubernetesにおけるroot権限の付与に相当します。

脆弱なコード例

YAML (Kubernetes RBAC) ❌ Bad 
# ClusterRoleBinding granting cluster-admin to a service account
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: app-admin-binding
subjects:
- kind: ServiceAccount
  name: my-app            # App SA with cluster-admin!
  namespace: default
roleRef:
  kind: ClusterRole
  name: cluster-admin      # Full cluster control!
  apiGroup: rbac.authorization.k8s.io

安全なコード例

YAML (Kubernetes RBAC) ✅ Good 
# Scoped Role with minimum required permissions
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: app-role
  namespace: my-app-ns
rules:
- apiGroups: [""]
  resources: ["configmaps"]
  verbs: ["get", "list"]     # Read-only, specific resources
  resourceNames: ["app-config"]  # Named resource only
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-role-binding
  namespace: my-app-ns
subjects:
- kind: ServiceAccount
  name: my-app
  namespace: my-app-ns
roleRef:
  kind: Role
  name: app-role
  apiGroup: rbac.authorization.k8s.io

対策チェックリスト

  • アプリケーションのサービスアカウントにcluster-adminをバインドしない。namespace-scopedのRoleを使用する
  • RBACルールでワイルドカード(*)権限を避ける。正確なverbとresourceを指定する
  • kubectl-who-canやrbac-policeなどのツールでRBACバインディングを定期的に監査する
  • 不要な場合はサービスアカウントトークンの自動マウントを無効化する

3️⃣ K03 - シークレット管理の失敗

Critical

概要

KubernetesのSecretsは、デフォルトではbase64エンコードされているだけで、暗号化されていません。機密データ(APIキー、データベースパスワード、TLS証明書)を平文のConfigMap、環境変数、暗号化されていないSecretsに保存すると、APIアクセス権を持つ誰にでも晒されます。また、etcdでの静止状態の暗号化が有効になっていない場合、Secretsはetcdで可視化されます。

リスク

漏洩したシークレットにより、攻撃者はデータベース、クラウドアカウント、外部サービスへアクセスできます。暗号化なしでetcdに保存されたSecretsは、etcdアクセス権を持つ誰でも読み取れます。環境変数のシークレットは、Pod仕様やプロセスリストで可視化されます。

脆弱なコード例

YAML (Kubernetes) ❌ Bad 
# Secret in plain ConfigMap — visible to anyone
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DB_PASSWORD: "SuperSecret123!"   # Plaintext password!
  API_KEY: "sk-live-abc123xyz"      # API key in ConfigMap!
---
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
  - name: app
    image: myapp:latest
    envFrom:
    - configMapRef:
        name: app-config    # All values as env vars!

安全なコード例

YAML (Kubernetes + External Secrets) ✅ Good 
# Use External Secrets Operator with a vault backend
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: app-secrets
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: app-secrets
  data:
  - secretKey: db-password
    remoteRef:
      key: secret/myapp/db
      property: password
---
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
  - name: app
    image: myapp@sha256:abc123...
    volumeMounts:
    - name: secrets
      mountPath: /etc/secrets
      readOnly: true          # Mount as read-only files
  volumes:
  - name: secrets
    secret:
      secretName: app-secrets

対策チェックリスト

  • etcdでKubernetes Secretsの静止状態の暗号化を有効にする
  • 外部シークレット管理(Vault、AWS Secrets Manager、External Secrets Operator)を使用する
  • 露出を減らすため、環境変数ではなくファイルとしてシークレットをマウントする
  • シークレットへのRBACアクセスを制限し、監査ログですべてのシークレットアクセスを監査する

4️⃣ K04 - クラスタレベルのポリシー実施の欠如

High

概要

クラスタレベルのポリシーがなければ、安全でないワークロードのデプロイを防ぐガードレールはありません。Pod Security Standards(PSS)、アドミッションコントローラ、OPA GatekeeperやKyvernoなどのポリシーエンジンは、すべてのnamespaceでセキュリティベースラインを実施するために不可欠です。

リスク

ポリシー実施がなければ、どの開発者でも特権付きコンテナをデプロイし、hostPathボリュームを使用し、セキュリティコントロールを無効化できます。単一の設定ミスのあるワークロードがクラスタ全体を侵害する可能性があります。手動レビューでは、すべての違反を検出するのに拡張できません。

脆弱なコード例

Bash ❌ Bad 
#!/bin/bash
# No admission control — any pod spec is accepted
# No Pod Security Standards configured

# Anyone can deploy a privileged container
kubectl run hacker-pod --image=alpine \
  --overrides='{"spec":{"containers":[{
    "name":"hack",
    "image":"alpine",
    "securityContext":{"privileged":true},
    "command":["nsenter","--target","1","--mount","--uts","--ipc","--net","--pid","--","bash"]
  }]}}'

# No policy blocks this — now has host root access!

安全なコード例

YAML (Kyverno ClusterPolicy) ✅ Good 
# Enforce Pod Security Standards with Kyverno
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: disallow-privileged
spec:
  validationFailureAction: Enforce
  background: true
  rules:
  - name: deny-privileged
    match:
      any:
      - resources:
          kinds: ["Pod"]
    validate:
      message: "Privileged containers are not allowed."
      pattern:
        spec:
          containers:
          - securityContext:
              privileged: "false"
  - name: require-run-as-non-root
    match:
      any:
      - resources:
          kinds: ["Pod"]
    validate:
      message: "Containers must run as non-root."
      pattern:
        spec:
          securityContext:
            runAsNonRoot: true

対策チェックリスト

  • すべての本番namespaceでPod Security Standards(Restricted)を実施する
  • ポリシーエンジン(Kyverno、OPA Gatekeeper)をenforceモードでデプロイする
  • ポリシーにより特権付きコンテナ、ホストnamespace、hostPathボリュームをブロックする
  • ポリシー違反を監査し、例外を定期的にレビューする

5️⃣ K05 - ネットワーク分離制御の欠如

High

概要

デフォルトでは、Kubernetesクラスタ内のすべてのPodは制限なしに相互通信できます。このフラットなネットワークアーキテクチャは、侵害されたPodが他のPod、サービス、さらにはKubernetes APIサーバーに到達できることを意味します。マイクロセグメンテーションを実装するには、NetworkPolicyが不可欠です。

リスク

ネットワーク分離がなければ、横方向の移動は簡単です。侵害されたフロントエンドPodは、バックエンドデータベース、内部サービス、メタデータAPI(169.254.169.254)に直接アクセスできます。これはネットワーク層での最小権限の原則に違反します。

脆弱なコード例

YAML (Kubernetes) ❌ Bad 
# No NetworkPolicy — all pods can communicate freely
apiVersion: v1
kind: Namespace
metadata:
  name: production
  # No default deny policy
  # No network policies at all
  # Any pod can reach:
  #   - Other pods in any namespace
  #   - Kubernetes API server
  #   - Cloud metadata API (169.254.169.254)
  #   - External internet

安全なコード例

YAML (Kubernetes NetworkPolicy) ✅ Good 
# Default deny all ingress and egress
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}
  policyTypes: ["Ingress", "Egress"]
---
# Allow only frontend to backend communication
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes: ["Ingress"]
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - port: 8080
      protocol: TCP

対策チェックリスト

  • すべてのnamespaceでingressとegressの両方にdefault-deny NetworkPolicyを適用する
  • サービス間で必要な通信パスのみをホワイトリスト化する
  • NetworkPolicyでクラウドメタデータAPI(169.254.169.254)へのPodアクセスをブロックする
  • NetworkPolicy実施をサポートするCNIプラグイン(Calico、Cilium)を使用する

6️⃣ K06 - 過度に露出したKubernetesコンポーネント

High

概要

Kubernetesコントロールプレーンコンポーネント(APIサーバー、etcd、kubelet、ダッシュボード)とアプリケーションサービスは、誤ってインターネットや信頼できないネットワークに露出する可能性があります。露出したダッシュボード、認証なしのkubelet、公開アクセス可能なAPIサーバーは一般的な攻撃ベクトルです。

リスク

露出したKubernetesコンポーネントは、クラスタ管理への直接アクセスを提供します。認証なしのkubelet APIはコンテナ実行を許可します。デフォルト認証情報を持つ露出したダッシュボードは、cluster-adminアクセスを付与します。公開etcdアクセスは、シークレットを含むすべてのクラスタデータを露出します。

脆弱なコード例

YAML (Kubernetes Service) ❌ Bad 
# Kubernetes Dashboard exposed via LoadBalancer
apiVersion: v1
kind: Service
metadata:
  name: kubernetes-dashboard
  namespace: kubernetes-dashboard
spec:
  type: LoadBalancer        # Public internet access!
  ports:
  - port: 443
    targetPort: 8443
  selector:
    k8s-app: kubernetes-dashboard
# Dashboard bound to cluster-admin service account
# No authentication required — full cluster access!

安全なコード例

YAML (Kubernetes) ✅ Good 
# Dashboard accessible only via kubectl proxy
apiVersion: v1
kind: Service
metadata:
  name: kubernetes-dashboard
  namespace: kubernetes-dashboard
spec:
  type: ClusterIP           # Internal only
  ports:
  - port: 443
    targetPort: 8443
  selector:
    k8s-app: kubernetes-dashboard
---
# Access via: kubectl proxy, then browse to:
# http://localhost:8001/api/v1/namespaces/kubernetes-dashboard/services/https:kubernetes-dashboard:/proxy/
# Or use kubectl port-forward for secure access

対策チェックリスト

  • Kubernetesダッシュボード、APIサーバー、etcdを公開インターネットに絶対に露出しない
  • 内部コンポーネントアクセスにはClusterIPサービスとkubectl proxy/port-forwardを使用する
  • すべてのKubernetesコンポーネント(kubelet、APIサーバー、etcd)で認証を有効にする
  • ファイアウォールルールまたは認可ネットワークを使用してAPIサーバーアクセスを信頼できるネットワークに制限する

7️⃣ K07 - 設定ミスと脆弱なクラスタコンポーネント

High

概要

Kubernetesクラスタは多くのコンポーネントで構成されています: APIサーバー、etcd、kubelet、kube-proxy、CoreDNS、サードパーティアドオン(Ingressコントローラ、サービスメッシュ、モニタリング)。設定ミスのある、またはパッチ未適用のコンポーネントは脆弱性を招きます。デフォルト設定は多くの場合強化されていません。

リスク

脆弱なクラスタコンポーネントは、リモートコード実行、権限昇格、またはサービス拒否のために悪用される可能性があります。kubelet、Ingressコントローラ、またはCNIプラグインのCVEは、コンテナエスケープとクラスタ乗っ取りにつながる可能性があります。時代遅れのコンポーネントは既知の脆弱性を蓄積します。

脆弱なコード例

YAML (Kubernetes) ❌ Bad 
# kubelet with insecure configuration
# /var/lib/kubelet/config.yaml
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
authentication:
  anonymous:
    enabled: true              # Anyone can access kubelet API!
  webhook:
    enabled: false             # No authorization!
authorization:
  mode: AlwaysAllow            # All requests permitted!
readOnlyPort: 10255            # Unauthenticated read port open!

安全なコード例

YAML (Kubernetes) ✅ Good 
# Hardened kubelet configuration
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
authentication:
  anonymous:
    enabled: false             # Deny anonymous access
  webhook:
    enabled: true              # Use API server for auth
  x509:
    clientCAFile: /etc/kubernetes/pki/ca.crt
authorization:
  mode: Webhook                # API server authorizes requests
readOnlyPort: 0                # Disable read-only port
protectKernelDefaults: true
eventRecordQPS: 5

対策チェックリスト

  • すべてのクラスタコンポーネントをセキュリティパッチが適用された最新の安定バージョンに保つ
  • kubeletで匿名認証と読み取り専用ポートを無効化する
  • CIS Kubernetes Benchmarkスキャンを定期的に実行する(kube-bench)
  • 相互TLSでetcdを強化し、コントロールプレーンノードのみにアクセスを制限する

8️⃣ K08 - クラスタからクラウドへの横方向移動

Critical

概要

クラウド環境(AWS、GCP、Azure)で実行されるKubernetesクラスタは、クラウドメタデータAPIとノードに付与されたIAMロールにアクセスできます。Podを侵害した攻撃者は、これらを使用してクラスタからクラウドアカウントへ権限を昇格し、S3バケット、データベース、その他のクラウドサービスにアクセスできます。

リスク

ノードレベルのIAMロールは、そのノード上のすべてのPodに継承されます。侵害されたPodは、メタデータAPI(169.254.169.254)にクエリしてクラウド認証情報を取得し、ノードロールが許可するあらゆるクラウドリソースにアクセスできます。これにより、Kubernetesから広範なクラウド環境への横方向の移動が可能になります。

脆弱なコード例

Terraform (AWS EKS) ❌ Bad 
# Node IAM role with excessive permissions
resource "aws_iam_role_policy_attachment" "node_admin" {
  role       = aws_iam_role.node_role.name
  policy_arn = "arn:aws:iam::aws:policy/AdministratorAccess"
  # All pods on this node inherit admin access!
}

# No IMDS restrictions — any pod can get node credentials
# curl http://169.254.169.254/latest/meta-data/iam/security-credentials/

安全なコード例

Terraform (AWS EKS with IRSA) ✅ Good 
# Use IAM Roles for Service Accounts (IRSA)
resource "aws_iam_role" "app_role" {
  name = "my-app-role"
  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect = "Allow"
      Principal = {
        Federated = aws_iam_openid_connect_provider.eks.arn
      }
      Action = "sts:AssumeRoleWithWebIdentity"
      Condition = {
        StringEquals = {
          "${aws_iam_openid_connect_provider.eks.url}:sub" =
            "system:serviceaccount:my-ns:my-app"
        }
      }
    }]
  })
}

# Pod-level scoped IAM via service account annotation
# Only this specific pod gets these specific permissions

対策チェックリスト

  • ノードレベルのロールではなく、PodレベルのIAM(AWSのIRSA、GCP/AzureのWorkload Identity)を使用する
  • NetworkPolicyでクラウドメタデータAPI(169.254.169.254)へのPodアクセスをブロックする
  • AWSでIMDSv2(ホップ制限 = 1)を実施して、Podがノードメタデータにアクセスできないようにする
  • 特定のリソースとアクションにスコープされた最小権限IAMポリシーを適用する

9️⃣ K09 - 破損した認証メカニズム

Critical

概要

Kubernetesは複数の認証メカニズムをサポートしています: 証明書、トークン、OIDC、Webhook認証。破損した認証には、静的トークンの使用、kubeconfigファイルの共有、証明書のローテーション不足、アイデンティティプロバイダーとの統合失敗が含まれます。サービスアカウントトークンの自動マウントは不必要な攻撃面を生み出します。

リスク

盗まれたまたは漏洩したkubeconfigファイルとサービスアカウントトークンは、クラスタへの直接アクセスを提供します。期限切れにならない静的トークンは永続的なアクセスを与えます。集中化されたアイデンティティ管理がなければ、退職したチームメンバーのアクセス取り消しは困難でエラーが発生しやすくなります。

脆弱なコード例

YAML (Kubernetes) ❌ Bad 
# Static token file for API server authentication
# /etc/kubernetes/token-auth-file.csv
# token,user,uid,"groups"
# abc123,admin,1,"system:masters"  # Static token, never expires!

# Pod with auto-mounted service account token
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  # automountServiceAccountToken defaults to true!
  # Token mounted at /var/run/secrets/kubernetes.io/serviceaccount/token
  containers:
  - name: app
    image: myapp:latest
    # App doesn't need K8s API access but has a token anyway

安全なコード例

YAML (Kubernetes) ✅ Good 
# Disable auto-mounted token for pods that don't need API access
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app
  namespace: my-app-ns
automountServiceAccountToken: false  # No auto-mount
---
# For pods that need API access, use bound tokens with expiry
apiVersion: v1
kind: Pod
metadata:
  name: api-consumer
spec:
  serviceAccountName: api-consumer-sa
  automountServiceAccountToken: false
  containers:
  - name: app
    image: myapp@sha256:abc123...
    volumeMounts:
    - name: token
      mountPath: /var/run/secrets/tokens
      readOnly: true
  volumes:
  - name: token
    projected:
      sources:
      - serviceAccountToken:
          expirationSeconds: 3600  # 1-hour expiry
          audience: api-server

対策チェックリスト

  • 静的トークンや証明書ではなく、アイデンティティプロバイダーとのOIDC統合を使用する
  • Kubernetes APIアクセスが不要なPodでautomountServiceAccountTokenを無効化する
  • 短い有効期限を持つバウンドサービスアカウントトークンを使用する
  • 証明書とトークンを定期的にローテーションする。退職したユーザーのアクセスを即座に取り消す

🔟 K10 - 不十分なログ記録と監視

Medium

概要

KubernetesはAPIサーバーリクエストの監査イベントを生成しますが、監査ログは無効化されているか、設定ミスが多くあります。適切なログ記録と監視がなければ、不正なシークレットアクセス、RBAC変更、コンテナエスケープなどの悪意ある活動は検出されません。コンテナとノードレベルでのランタイムセキュリティ監視が不可欠です。

リスク

監査ログがなければ、攻撃者は検出されずに活動します。バックドアサービスアカウントの作成、シークレットの抽出、ワークロードの変更を記録なしで行えます。何が起こったか、いつ、誰によってかを判断する監査証跡がない場合、インシデント対応は深刻に損なわれます。

脆弱なコード例

YAML (Kubernetes) ❌ Bad 
# API server with no audit logging configured
# kube-apiserver flags:
#   --audit-log-path=""           # No audit log!
#   --audit-policy-file=""        # No audit policy!

# No runtime security monitoring
# No alerting on suspicious activities
# Container logs not collected centrally
# Default log retention — logs lost on pod restart

安全なコード例

YAML (Kubernetes Audit Policy) ✅ Good 
# Comprehensive audit policy
apiVersion: audit.k8s.io/v1
kind: Policy
rules:
# Log all secret access at Metadata level
- level: Metadata
  resources:
  - group: ""
    resources: ["secrets"]
# Log RBAC changes at RequestResponse level
- level: RequestResponse
  resources:
  - group: rbac.authorization.k8s.io
    resources: ["clusterroles", "clusterrolebindings", "roles", "rolebindings"]
# Log pod exec/attach (potential attacks)
- level: Request
  resources:
  - group: ""
    resources: ["pods/exec", "pods/attach", "pods/portforward"]
# Default: log at Metadata level
- level: Metadata

対策チェックリスト

  • 包括的な監査ポリシーでKubernetes API監査ログを有効にする
  • コンテナレベルの脅威検出のためにランタイムセキュリティ監視(Falco、Tetragon)をデプロイする
  • 監査ログとコンテナログを改ざん防止ストレージを持つ集中型SIEMに転送する
  • 高リスクイベントのアラートを作成する: シークレットアクセス、RBAC変更、Pod exec、権限昇格
