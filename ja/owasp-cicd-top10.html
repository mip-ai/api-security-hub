OWASP Top 10 CI/CDセキュリティリスク

CI/CDパイプラインにおける10の最重要セキュリティリスクとその対策方法。

CI/CDセキュリティ Top 10リスクとは？

OWASP Top 10 CI/CDセキュリティリスクは、継続的インテグレーション（CI）と継続的デリバリー（CD）パイプラインにおける最も重大なセキュリティリスクを特定しています。CI/CDパイプラインは、ソースコード、シークレット、本番環境への直接アクセスを持つため、高価値のターゲットです。このガイドでは、フロー制御、ID管理、依存関係攻撃、パイプラインポイズニングなどを取り上げます。

1️⃣ CICD-SEC-1 - 不十分なフロー制御メカニズム

Critical

概要

不十分なフロー制御メカニズムにより、攻撃者は適切なレビューや承認ゲートなしに悪意のあるコードをCI/CDパイプラインにプッシュできます。これには、ブランチ保護ルールのバイパス、必要な承認の欠如、本番環境へのデプロイをトリガーできるユーザーの制限不足が含まれます。

リスク

適切なフロー制御がないと、侵害された1つの開発者アカウントや悪意のあるインサイダーが、コードレビュー、セキュリティスキャン、承認プロセスをバイパスして、コードを直接本番環境にプッシュできます。これにより、バックドア、データ流出コード、または破壊的なペイロードのデプロイにつながる可能性があります。

脆弱なコード例

YAML (GitHub Actions) ❌ 悪い例 
# No branch protection — anyone can push directly to main
name: Deploy
on:
  push:
    branches: [main]  # Triggers on any push to main
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - run: |
        # Deploys immediately — no approval gate!
        kubectl apply -f deploy.yaml
        kubectl rollout status deployment/myapp

安全なコード例

YAML (GitHub Actions) ✅ 良い例 
name: Deploy
on:
  push:
    branches: [main]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - run: npm test
  deploy:
    needs: test
    runs-on: ubuntu-latest
    environment: production  # Requires manual approval
    steps:
    - uses: actions/checkout@v4
    - run: |
        kubectl apply -f deploy.yaml
        kubectl rollout status deployment/myapp

緩和チェックリスト

  • mainへのマージ前にPRレビューを必要とするブランチ保護ルールを強制する
  • 本番デプロイメントに必須の承認者を持つ環境保護ルールを使用する
  • マージ前に合格する必要があるステータスチェック（テスト、セキュリティスキャン）を実装する
  • 保護されたブランチへのプッシュと本番デプロイメントのトリガーを行えるユーザーを制限する

2️⃣ CICD-SEC-2 - 不十分なIDおよびアクセス管理

Critical

概要

CI/CDシステムには、人間のユーザー、サービスアカウント、ボットトークン、マシンIDなど、複数のIDが関与しています。不十分なIAMにより、リポジトリ、パイプライン、デプロイターゲットへの過度に許可的なアクセスが許されます。共有資格情報、古いアカウント、MFAの欠如がリスクを増大させます。

リスク

過剰な権限を持つ侵害されたCI/CD IDは、パイプラインの変更、シークレットへのアクセス、ビルドアーティファクトの改ざん、本番環境へのデプロイが可能です。共有サービスアカウントは、誰がアクションを実行したかの監査を不可能にします。

脆弱なコード例

YAML (GitHub Actions) ❌ 悪い例 
# Over-permissive workflow with admin token
name: CI
on: push
permissions: write-all  # Full permissions to everything!
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - run: |
        # Using org-wide PAT with admin access
        curl -H "Authorization: token ${{ secrets.ADMIN_PAT }}" \
          https://api.github.com/repos/org/other-repo/contents/

安全なコード例

YAML (GitHub Actions) ✅ 良い例 
name: CI
on: push
permissions:
  contents: read    # Minimum required permissions
  packages: write   # Only what's needed
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: actions/create-github-app-token@v1  # Scoped app token
      id: app-token
      with:
        app-id: ${{ vars.APP_ID }}
        private-key: ${{ secrets.APP_PRIVATE_KEY }}
        repositories: other-repo  # Scoped to specific repo

緩和チェックリスト

  • すべてのワークフロートークンとサービスアカウントに最小権限の原則を適用する
  • 長期間有効なPATの代わりにGitHub AppsまたはOIDCフェデレーションを使用する
  • CI/CDアクセスを持つすべての人間のアカウントにMFAを強制する
  • 資格情報を定期的に監査・ローテーションし、古いアカウントとトークンを削除する

3️⃣ CICD-SEC-3 - 依存関係チェーンの悪用

Critical

概要

CI/CDパイプラインは外部パッケージレジストリ（npm、PyPI、Maven、Docker Hub）から依存関係を取得します。攻撃者は、依存関係の混乱、タイポスクワッティング、侵害されたメンテナーアカウント、悪意のあるパッケージを通じてこれを悪用します。単一の汚染された依存関係がビルド中に任意のコードを実行できます。

リスク

悪意のある依存関係はインストール時にコードを実行し（postinstallスクリプト、setup.py）、シークレットの窃取、ビルドアーティファクトへのバックドア注入、または永続化を確立します。SolarWindsやCodecovのようなサプライチェーン攻撃は、その壊滅的な影響を実証しています。

脆弱なコード例

Jenkinsfile ❌ 悪い例 
pipeline {
  agent any
  stages {
    stage('Build') {
      steps {
        // No lockfile verification, no integrity checks
        sh 'npm install'           // Fetches latest — could be compromised!
        sh 'pip install -r requirements.txt'  // No hash verification
      }
    }
  }
}

安全なコード例

Jenkinsfile ✅ 良い例 
pipeline {
  agent any
  stages {
    stage('Build') {
      steps {
        // Use lockfile with integrity verification
        sh 'npm ci'                   // Uses package-lock.json
        sh 'npm audit --audit-level=high'
        // Python: verify hashes from lockfile
        sh 'pip install --require-hashes -r requirements.lock'
      }
    }
    stage('SCA Scan') {
      steps {
        // Software Composition Analysis
        sh 'trivy fs --scanners vuln,secret .'
      }
    }
  }
}

緩和チェックリスト

  • 常にロックファイル（package-lock.json、Pipfile.lock）を使用し、整合性ハッシュを検証する
  • 依存関係の混乱を防ぐために、ネームスペーススコーピングを持つプライベート/内部パッケージレジストリを設定する
  • すべてのビルドパイプラインでソフトウェア構成分析（SCA）ツールを実行する
  • 依存関係を正確なバージョンにピン留めし、マージ前にすべての依存関係の更新をレビューする

4️⃣ CICD-SEC-4 - パイプライン実行のポイズニング（PPE）

Critical

概要

パイプライン実行のポイズニング（PPE）は、攻撃者がCI/CDパイプライン定義を変更したり、パイプラインコンテキスト内で実行される悪意のあるコードを注入できる場合に発生します。これは、ブランチ内のパイプライン設定ファイルの操作、フォークからのプルリクエスト、または共有パイプラインテンプレートの変更を通じて起こり得ます。

リスク

パイプライン設定を変更できる攻撃者は、パイプラインが利用可能なすべてのシークレット、資格情報、権限にアクセスできます。信頼された実行コンテキスト内で、シークレットの流出、ビルド出力の改ざん、または悪意のあるコードのデプロイが可能になります。

脆弱なコード例

YAML (GitHub Actions) ❌ 悪い例 
# Runs pipeline from fork PRs with access to secrets
name: CI
on:
  pull_request_target:  # Runs in base repo context with secrets!
    branches: [main]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
      with:
        ref: ${{ github.event.pull_request.head.sha }}  # Checks out fork code!
    - run: make build  # Fork's Makefile executes with secrets access
      env:
        DEPLOY_KEY: ${{ secrets.DEPLOY_KEY }}

安全なコード例

YAML (GitHub Actions) ✅ 良い例 
# Separate workflows: untrusted build + trusted deploy
name: CI
on:
  pull_request:  # No secret access for PR builds
    branches: [main]
permissions:
  contents: read
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4  # Checks out merge commit (safe)
    - run: npm ci && npm test   # No secrets needed for build/test
    - uses: actions/upload-artifact@v4
      with:
        name: build-output
        path: dist/

緩和チェックリスト

  • pull_request_targetでPRヘッドコードのチェックアウトとシークレットアクセスを組み合わせて使用しない
  • ビルド（非信頼）とデプロイ（信頼）のステージを異なるワークフローに分離する
  • フォークPRワークフローに承認を要求し、シークレットの公開を制限する
  • コントリビューターが変更できない不変のパイプライン定義を使用する

5️⃣ CICD-SEC-5 - 不十分なパイプラインベースのアクセス制御（PBAC）

High

概要

パイプラインベースのアクセス制御（PBAC）は、パイプラインがアクセスできるリソースを管理します：クラウドアカウント、Kubernetesクラスタ、データベース、内部サービスなど。不十分なPBACは、パイプラインが必要以上に広いアクセス権を持つことを意味し、最小権限の原則に違反します。

リスク

過度に権限を持つパイプラインは、意図した範囲をはるかに超えたリソースにアクセスするために悪用される可能性があります。軽微なサービスの侵害されたビルドパイプラインが、本番データベースへのアクセス、インフラストラクチャの変更、または他の環境へのピボットに使用される恐れがあります。

脆弱なコード例

YAML (GitLab CI) ❌ 悪い例 
# Pipeline with admin-level cloud credentials
deploy:
  stage: deploy
  script:
    - aws configure set aws_access_key_id $AWS_ACCESS_KEY
    - aws configure set aws_secret_access_key $AWS_SECRET_KEY
    # This key has AdministratorAccess policy!
    - aws s3 sync dist/ s3://my-bucket/
    - aws ecs update-service --cluster prod --service myapp
    # Same credentials could access ANY AWS resource

安全なコード例

YAML (GitHub Actions) ✅ 良い例 
name: Deploy
on:
  push:
    branches: [main]
permissions:
  id-token: write  # For OIDC
  contents: read
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
    - uses: aws-actions/configure-aws-credentials@v4
      with:
        role-to-assume: arn:aws:iam::123456:role/deploy-s3-only
        # Scoped role: only s3:PutObject on specific bucket
        aws-region: us-east-1
    - run: aws s3 sync dist/ s3://my-bucket/

緩和チェックリスト

  • CI/CDでの静的な資格情報の代わりに、クラウドアクセスにOIDCフェデレーションを使用する
  • パイプラインごとにIAMロールをスコープし、そのパイプラインが必要とするリソースのみに限定する
  • 環境ごと（ステージングvs.本番）にデプロイメントロールを分離する
  • パイプラインのアクセスパターンを監査し、スコープ外のリソースアクセスにアラートを出す

6️⃣ CICD-SEC-6 - 不十分な資格情報の衛生管理

Critical

概要

CI/CDパイプラインは多数の資格情報を扱います：APIキー、クラウドトークン、レジストリパスワード、SSHキー、署名証明書など。資格情報の衛生管理が不十分な場合、パイプラインファイルへのシークレットのハードコード、ログへのシークレットの出力、暗号化されていないシークレットストレージの使用、資格情報のローテーション未実施が含まれます。

リスク

漏洩したCI/CD資格情報は、最も一般的な初期アクセスベクターの一つです。ビルドログに露出した、リポジトリにコミットされた、または暗号化なしで保存されたシークレットは、攻撃者に本番システム、クラウドアカウント、アーティファクトレジストリへの直接アクセスを提供します。

脆弱なコード例

Bash (CI Script) ❌ 悪い例 
#!/bin/bash
# Secrets hardcoded and leaked in logs
export DOCKER_PASSWORD="MyS3cret!"         # Hardcoded!
echo "Logging in with $DOCKER_PASSWORD"  # Printed to logs!

docker login -u admin -p "$DOCKER_PASSWORD" registry.example.com
docker push registry.example.com/myapp:latest

# AWS credentials in environment — visible in process listing
export AWS_SECRET_ACCESS_KEY="wJalrXUtnFEMI/K7MDENG/bPxRfiCY"
aws s3 cp artifact.zip s3://releases/

安全なコード例

YAML (GitHub Actions) ✅ 良い例 
name: Publish
on:
  push:
    tags: ['v*']
jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: docker/login-action@v3  # Handles credentials securely
      with:
        registry: registry.example.com
        username: ${{ secrets.DOCKER_USER }}   # Masked in logs
        password: ${{ secrets.DOCKER_TOKEN }}  # Short-lived token
    - run: |
        docker build -t registry.example.com/myapp:${{ github.ref_name }} .
        docker push registry.example.com/myapp:${{ github.ref_name }}

緩和チェックリスト

  • パイプライン定義、スクリプト、ソースコードにシークレットをハードコードしない
  • 自動ログマスキング機能を持つCI/CDプラットフォームのシークレット管理を使用する
  • すべてのCI/CD資格情報を定期的にローテーションし、可能な限り短期トークンを使用する
  • リポジトリとビルドログで誤って露出したシークレットをスキャンする

7️⃣ CICD-SEC-7 - 安全でないシステム設定

High

概要

CI/CDシステム（Jenkins、GitLab、GitHub Actionsランナー）は、安全でないデフォルト設定で実行されていることが多くあります。これには、古いソフトウェアバージョン、露出した管理インターフェース、無効化されたセキュリティ機能、過度に許可的なネットワークアクセス、プロジェクト間で共有されるセルフホストランナーが含まれます。

リスク

誤設定されたCI/CDインフラストラクチャは、ビルド環境への不正アクセス、シークレットの傍受、内部ネットワークへのピボットに悪用される可能性があります。共有のセルフホストランナーでは、侵害された1つのワークフローが他のプロジェクトにも影響を与えるクロスプロジェクト攻撃が可能になります。

脆弱なコード例

Groovy (Jenkins) ❌ 悪い例 
// Jenkins with insecure configuration
// - Script console enabled without auth
// - Agent-to-controller access unrestricted
// - Outdated plugins with known CVEs

pipeline {
  agent any  // Runs on any available agent — no isolation
  stages {
    stage('Build') {
      steps {
        // Running as root on shared agent
        sh 'whoami'  // root
        sh 'docker build -t myapp .'
      }
    }
  }
}

安全なコード例

Groovy (Jenkins) ✅ 良い例 
pipeline {
  agent {
    kubernetes {  // Ephemeral, isolated pod per build
      yaml """
        apiVersion: v1
        kind: Pod
        spec:
          securityContext:
            runAsNonRoot: true
            runAsUser: 1000
          containers:
          - name: builder
            image: builder:1.2.3
            securityContext:
              allowPrivilegeEscalation: false
              readOnlyRootFilesystem: true
      """
    }
  }
  stages {
    stage('Build') {
      steps {
        container('builder') {
          sh 'make build'
        }
      }
    }
  }
}

緩和チェックリスト

  • 各ビルド後に破棄されるエフェメラルで分離されたビルド環境（コンテナ/VM）を使用する
  • CI/CDプラットフォームとすべてのプラグインを最新バージョンに更新する
  • ビルド環境からのネットワークアクセスを制限し、不要な送信トラフィックをブロックする
  • ビルドエージェントを最小限のケイパビリティと読み取り専用ファイルシステムで非rootとして実行する

8️⃣ CICD-SEC-8 - サードパーティサービスの管理されていない使用

High

概要

CI/CDパイプラインは、コード品質ツール、セキュリティスキャナー、通知システム、デプロイメントプラットフォームなどのサードパーティサービスと統合されることが多くあります。これらの統合にはアクセストークンと権限が付与され、信頼チェーンが形成されます。管理されていない使用とは、サードパーティサービスが何にアクセスし何ができるかの可視性がないことを意味します。

リスク

侵害されたサードパーティサービス（Codecov侵害のような）は、ソースコード、シークレット、ビルドアーティファクトにアクセスできます。ガバナンスがないと、チームは未知のサービスに過剰な権限を付与し、サプライチェーンに不可視の攻撃ベクターを作り出す可能性があります。

脆弱なコード例

YAML (GitHub Actions) ❌ 悪い例 
name: CI
on: push
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - run: npm test
    # Unknown third-party action with full repo access
    - uses: random-org/code-analysis@main  # Unpinned, unvetted!
      with:
        token: ${{ secrets.GITHUB_TOKEN }}   # Full token access!
    # Uploading coverage to external service with repo token
    - run: |
        bash <(curl -s https://example.com/uploader.sh)  # Remote script!

安全なコード例

YAML (GitHub Actions) ✅ 良い例 
name: CI
on: push
permissions:
  contents: read
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - run: npm test
    # Vetted action pinned to SHA
    - uses: github/codeql-action/analyze@8a470fddafa5cbc14  # Pinned SHA
    # Upload via official CLI tool, not remote scripts
    - uses: codecov/codecov-action@e28ff129e5465c2c0dcc6f003fc735cb6ae0c673
      with:
        token: ${{ secrets.CODECOV_TOKEN }}  # Scoped token
        fail_ci_if_error: true

緩和チェックリスト

  • CI/CDと統合されたすべてのサードパーティサービスのインベントリを維持する
  • サードパーティアクションをSHAハッシュにピン留めし、採用前に監査する
  • パイプラインでリモートスクリプト（curl | bash）を実行しない
  • サードパーティ統合に付与された権限を定期的にレビューし最小化する

9️⃣ CICD-SEC-9 - 不適切なアーティファクト整合性検証

High

概要

ビルドアーティファクト（コンテナイメージ、バイナリ、パッケージ）はCI/CDパイプラインを通じて本番環境に配信されます。整合性検証がないと、アーティファクトはビルド中、転送中、アーティファクトレジストリ内、デプロイ時のいずれのポイントでも改ざんされる可能性があります。これにより、コードから本番までの信頼チェーンが破壊されます。

リスク

改ざんされたアーティファクトには、バックドア、マルウェア、または変更されたロジックが含まれる可能性があります。署名と検証がないと、ビルド後にアーティファクトが変更されたかどうかを検出する方法がありません。攻撃者はレジストリ内の正規イメージを置き換えたり、転送中のアーティファクトを傍受できます。

脆弱なコード例

Bash ❌ 悪い例 
#!/bin/bash
# Build and deploy without any integrity checks
docker build -t myregistry.com/app:latest .
docker push myregistry.com/app:latest

# On deployment side — no verification
docker pull myregistry.com/app:latest  # Could be tampered!
docker run myregistry.com/app:latest   # Tag is mutable!

安全なコード例

YAML (GitHub Actions) ✅ 良い例 
name: Build and Sign
on:
  push:
    tags: ['v*']
permissions:
  id-token: write
  packages: write
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - run: |
        # Build with immutable tag (SHA)
        docker build -t myregistry.com/app:${{ github.sha }} .
        docker push myregistry.com/app:${{ github.sha }}
    - uses: sigstore/cosign-installer@v3
    - run: |
        # Sign image with keyless signing (Sigstore)
        cosign sign myregistry.com/app:${{ github.sha }}
        # Generate and attach SBOM
        syft myregistry.com/app:${{ github.sha }} -o spdx-json > sbom.json
        cosign attest --predicate sbom.json myregistry.com/app:${{ github.sha }}

緩和チェックリスト

  • Cosign/SigstoreまたはNotaryを使用してすべてのビルドアーティファクト（イメージ、バイナリ）に署名する
  • アドミッションコントローラー（Kyverno、OPA Gatekeeper）でデプロイ時に署名を検証する
  • :latestのような変更可能なタグの代わりに不変のタグ（SHAダイジェスト）を使用する
  • ソースからアーティファクトまでの検証可能なチェーンを確立するためにSLSAプロベナンスを実装する

🔟 CICD-SEC-10 - 不十分なログ記録と可視性

Medium

概要

CI/CD環境は重要なセキュリティイベントを生成します：パイプラインの実行、設定変更、シークレットアクセス、デプロイメント活動など。包括的なログ記録と監視がないと、パイプライン内の悪意のある活動は検出されず、インシデント対応が著しく妨げられます。

リスク

CI/CD活動の可視性がないと、攻撃者はアラートをトリガーすることなく、パイプラインの変更、シークレットの流出、アーティファクトの改ざんが可能です。監査証跡の欠如により、侵害の範囲と影響を特定することが不可能になります。

脆弱なコード例

Bash ❌ 悪い例 
#!/bin/bash
# Pipeline with no logging or audit trail
echo "Starting deployment..."
kubectl apply -f deploy.yaml
echo "Done."

# No record of: who triggered this, what changed,
# which image was deployed, what secrets were accessed
# Build logs expire after 30 days with no archival

安全なコード例

YAML (GitHub Actions) ✅ 良い例 
name: Audited Deploy
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - name: Log deployment metadata
      run: |
        echo "=== Deployment Audit Log ==="
        echo "Triggered by: ${{ github.actor }}"
        echo "Commit: ${{ github.sha }}"
        echo "Ref: ${{ github.ref }}"
        echo "Workflow: ${{ github.workflow }}"
        echo "Run ID: ${{ github.run_id }}"
    - name: Deploy with audit
      run: |
        kubectl apply -f deploy.yaml 2>&1 | tee deploy.log
        # Send audit event to SIEM
        curl -X POST "${{ secrets.SIEM_WEBHOOK }}" \
          -d '{"event":"deploy","actor":"${{ github.actor }}","sha":"${{ github.sha }}"}'

緩和チェックリスト

  • すべてのCI/CDプラットフォーム活動（パイプライン変更、シークレットアクセス、ユーザーアクション）の監査ログを有効化する
  • CI/CD監査ログを改ざん防止ストレージを持つ集中型SIEMに転送する
  • 不審な活動に対するアラートを作成する：異常なデプロイ時間、設定変更、新しいサービス統合
  • コンプライアンスとフォレンジック分析のためにビルドとデプロイメントのログを保持する
