OWASP Cloud-Native Application Security Top 10

クラウドネイティブアプリケーションにおける10の最重要セキュリティリスクとその対策方法。

Cloud-Native Application Security Top 10とは？

OWASP Cloud-Native Application Security Top 10は、Kubernetes、Docker、サーバーレスアーキテクチャなどのプラットフォーム上で稼働するクラウドネイティブアプリケーションにおける最も顕著なセキュリティリスクを特定しています。誤設定、サプライチェーンリスク、シークレット管理など、クラウドネイティブスタック全体にわたる問題を網羅しています。

1️⃣ CNAS-1 - 安全でないクラウド、コンテナ、またはオーケストレーション設定

Critical

概要

誤設定されたクラウドサービス、コンテナ、オーケストレーターは、クラウドネイティブ侵害の主要な原因です。これには、コンテナをrootとして実行すること、デフォルト設定の使用、Kubernetes APIサーバーの公開、クラウドリソースの監査ログ有効化失敗などが含まれます。

リスク

攻撃者は誤設定を悪用してクラスタの完全な制御を取得し、コンテナをエスケープし、クラウドメタデータサービスにアクセスしたり、インフラストラクチャ全体にピボットすることができます。誤設定されたS3バケットや公開されたKubernetesダッシュボード一つで、大規模なデータ侵害につながった事例があります。

脆弱なコード例

Dockerfile ❌ Bad 
# Running container as root with no resource limits
FROM ubuntu:latest
RUN apt-get update && apt-get install -y curl wget
COPY app /app
# No USER directive — runs as root!
CMD ["/app/server"]

安全なコード例

Dockerfile ✅ Good 
FROM gcr.io/distroless/static:nonroot
COPY --chown=65534:65534 app /app
USER 65534:65534
EXPOSE 8080
ENTRYPOINT ["/app/server"]

対策チェックリスト

  • 最小限のdistrolessベースイメージを使用し、コンテナをrootで実行しない
  • KubernetesでPod Security Standards（restricted）を有効化して強制する
  • CheckovやtfsecなどのツールでInfrastructure as Code（Terraform、Helm）をスキャンする
  • クラウドメタデータエンドポイント（169.254.169.254）およびKubernetes APIサーバーへのアクセスを制限する

2️⃣ CNAS-2 - インジェクションの欠陥（アプリケーション層、クラウドイベント、クラウドサービス）

Critical

概要

クラウドネイティブアプリケーションは、従来のHTTPリクエストだけでなく、クラウドイベント（SQS、Pub/Sub、EventBridge）、サーバーレストリガー、サービス間通信からも入力を受け取ります。これらのベクターのいずれかでインジェクション欠陥が発生すると、コマンド実行、データ流出、または権限昇格につながる可能性があります。

リスク

クラウドイベントによってトリガーされるサーバーレス関数は、検証なしで信頼できないデータを処理する可能性があり、OSコマンドインジェクション、NoSQLインジェクション、またはイベント駆動型SSRFにつながります。攻撃者はメッセージキューやイベントバスを汚染して、下流のサービスを侵害することができます。

脆弱なコード例

Python ❌ Bad 
# Lambda function processing S3 event without sanitizing filename
import os

def handler(event, context):
    bucket = event['Records'][0]['s3']['bucket']['name']
    key = event['Records'][0]['s3']['object']['key']
    # Command injection via crafted filename!
    os.system(f"aws s3 cp s3://{bucket}/{key} /tmp/{key}")

安全なコード例

Python ✅ Good 
import boto3, re
from urllib.parse import unquote_plus

def handler(event, context):
    s3 = boto3.client('s3')
    bucket = event['Records'][0]['s3']['bucket']['name']
    key = unquote_plus(event['Records'][0]['s3']['object']['key'])

    # Validate key against allowlist pattern
    if not re.match(r'^[\w\-./]+$', key):
        raise ValueError(f"Invalid S3 key: {key}")

    # Use SDK instead of shell commands
    s3.download_file(bucket, key, f'/tmp/{key.split("/")[-1]}')

対策チェックリスト

  • クラウドイベント、キュー、トリガーからのすべての入力を検証およびサニタイズする
  • シェルコマンドの代わりにSDK/API呼び出しを使用する（os.system、shell=Trueのsubprocessを避ける）
  • イベントペイロードに入力検証スキーマを適用する（JSON Schema、protobuf）
  • サーバーレス関数からのSSRFを防ぐためにエグレスフィルタリングを実装する

3️⃣ CNAS-3 - 不適切な認証と認可

Critical

概要

クラウドネイティブ環境は、複数のアイデンティティレイヤーを含みます：クラウドIAM、Kubernetes RBAC、サービスメッシュmTLS、アプリケーションレベルの認証。いずれかのレイヤーで誤設定されたり過度に許可的なポリシーがあると、横展開、権限昇格、機密リソースへの不正アクセスを許してしまいます。

リスク

ポッドに割り当てられた過度に許可的なIAMロールは、ワークロードにクラウドアカウント全体へのアクセスを許可する可能性があります。サービス間認証の欠如により、侵害されたポッドが他のサービスになりすますことができます。

脆弱なコード例

YAML (Kubernetes) ❌ Bad 
# Overly permissive ClusterRoleBinding
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: default-admin
subjects:
- kind: ServiceAccount
  name: default          # Default SA — shared by all pods!
  namespace: default
roleRef:
  kind: ClusterRole
  name: cluster-admin    # Full cluster access!
  apiGroup: rbac.authorization.k8s.io

安全なコード例

YAML (Kubernetes) ✅ Good 
# Dedicated ServiceAccount with minimal Role
apiVersion: v1
kind: ServiceAccount
metadata:
  name: order-service
  namespace: production
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456:role/order-svc
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: order-service-role
  namespace: production
rules:
- apiGroups: [""]
  resources: ["configmaps"]
  verbs: ["get"]

対策チェックリスト

  • ワークロードごとに最小権限RBACを持つ専用のServiceAccountを使用する
  • ノードレベルの資格情報の代わりにIAM Roles for Service Accounts（IRSA/Workload Identity）を実装する
  • すべてのサービス間通信で相互TLS（mTLS）を強制する
  • RBACバインディングを定期的に監査し、使用されていない権限を削除する

4️⃣ CNAS-4 - CI/CDパイプラインとソフトウェアサプライチェーンの欠陥

High

概要

CI/CDパイプラインは、本番環境への書き込みアクセスを持つため、価値の高いターゲットです。ビルドパイプラインの侵害により、サプライチェーン攻撃、悪意のあるコードの注入、バックドア付きイメージのデプロイが可能になります。安全でないパイプライン設定、汚染されたベースイメージ、署名されていないアーティファクトがこのリスクに貢献します。

リスク

侵害されたCI/CDパイプラインは、すべての環境に悪意のあるコードをデプロイし、パイプライン変数に保存されたシークレットを窃取したり、コンテナイメージを改ざんすることができます。SolarWindsスタイルの攻撃は、サプライチェーン侵害の壊滅的な影響を示しています。

脆弱なコード例

YAML (GitHub Actions) ❌ Bad 
# Insecure CI pipeline — unpinned actions, no image signing
name: Deploy
on: push
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@main   # Unpinned — could be compromised!
    - run: |
        docker build -t myapp:latest .
        docker push myregistry/myapp:latest  # No signing!
        kubectl apply -f deploy.yaml         # No admission control!

安全なコード例

YAML (GitHub Actions) ✅ Good 
name: Secure Deploy
on: push
permissions:
  id-token: write
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@a5ac7e51b41094c92402da3b24376905380afc29  # Pinned SHA
    - run: |
        docker build -t myregistry/myapp:${{ github.sha }} .
        cosign sign myregistry/myapp:${{ github.sha }}  # Sign image
    - run: trivy image myregistry/myapp:${{ github.sha }} --exit-code 1
    - run: kubectl apply -f deploy.yaml  # Admission controller verifies signatures

対策チェックリスト

  • CI/CDアクションと依存関係を正確なSHAハッシュにピン留めし、ブランチやタグは使用しない
  • Cosign/Notaryでコンテナイメージに署名し、アドミッションで署名検証を強制する
  • デプロイ前にイメージの脆弱性をスキャンする（Trivy、Grype）
  • SLSAフレームワークを使用してビルドアーティファクトの出自と整合性を保証する

5️⃣ CNAS-5 - 安全でないシークレット保管

Critical

概要

APIキー、データベース資格情報、TLS証明書などのシークレットは、ソースコードにハードコードされたり、平文のConfigMapに保存されたり、コンテナイメージに埋め込まれることが多くあります。Kubernetes Secretsはデフォルトでbase64エンコードされているだけで暗号化されていないため、最小限の保護しか提供しません。

リスク

露出したシークレットは、攻撃者にデータベース、クラウドアカウント、サードパーティサービスへの直接アクセスを与える可能性があります。git履歴、コンテナレイヤー、環境変数内のシークレットは容易に発見され悪用されます。

脆弱なコード例

YAML (Kubernetes) ❌ Bad 
# Secrets in plaintext environment variables
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    env:
    - name: DB_PASSWORD
      value: "SuperSecret123!"   # Plaintext in manifest!
    - name: AWS_SECRET_KEY
      value: "AKIA..."             # Cloud credentials in YAML!

安全なコード例

YAML (Kubernetes) ✅ Good 
# Use External Secrets Operator with a vault backend
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: my-app-secrets
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: my-app-secrets
  data:
  - secretKey: db-password
    remoteRef:
      key: secret/data/myapp
      property: db-password

対策チェックリスト

  • シークレットをソースコード、Dockerfile、平文のConfigMapに保存しない
  • 自動ローテーション機能を持つシークレットマネージャー（Vault、AWS Secrets Manager、GCP Secret Manager）を使用する
  • Kubernetes Secretsの保存時暗号化を有効化する（KMSプロバイダー）
  • リポジトリとイメージで漏洩したシークレットをスキャンする（git-secrets、TruffleHog、Trivy）

6️⃣ CNAS-6 - 過度に許可的または安全でないネットワークポリシー

High

概要

デフォルトでは、Kubernetesはすべてのポッドが制限なく相互に通信することを許可します。欠落しているか過度に許可的なNetworkPolicies、セキュリティグループ、ファイアウォールルールは、クラスタ内およびクラウドサービス間の横展開を可能にします。

リスク

ネットワークセグメンテーションがないと、侵害されたポッドがクラスタ内の他のサービスに到達し、データベースに直接アクセスしたり、外部エンドポイントにデータを流出させることができます。フラットなネットワークは、あらゆる侵害の影響範囲を増幅させます。

脆弱なコード例

YAML (Kubernetes) ❌ Bad 
# No NetworkPolicy — all pods can talk to everything
apiVersion: v1
kind: Namespace
metadata:
  name: production
# No NetworkPolicy resources defined
# All ingress and egress traffic is allowed by default

安全なコード例

YAML (Kubernetes) ✅ Good 
# Default-deny all traffic, then allow only what's needed
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}
  policyTypes: ["Ingress", "Egress"]
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-api
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api-server
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - port: 8080

対策チェックリスト

  • すべてのネームスペースにデフォルト拒否のNetworkPoliciesを実装する
  • ラベルセレクタを使用してサービス間の必要なトラフィックのみを許可する
  • データ流出とC2通信を防ぐためにエグレストラフィックを制限する
  • NetworkPolicy強制をサポートするCNIプラグイン（Calico、Cilium）を使用する

7️⃣ CNAS-7 - 既知の脆弱性を持つコンポーネントの使用

High

概要

クラウドネイティブアプリケーションは、オープンソースのベースイメージ、ライブラリ、Kubernetesオペレーターに大きく依存しています。既知のCVEを持つ古いまたはパッチ未適用のコンポーネントを実行すると、文書化されたエクスプロイトにアプリケーションを晒すことになります。コンテナイメージには数百のパッケージが含まれることが多く、それぞれが潜在的な脆弱性となります。

リスク

ベースイメージの既知の脆弱性（例：Log4Shell、OpenSSLのバグ）は、公開されているツールを使用して悪用できます。攻撃者は脆弱なバージョンを実行しているコンテナを積極的にスキャンしています。パッチ未適用のライブラリ一つで、ワークロード全体が侵害される可能性があります。

脆弱なコード例

Dockerfile ❌ Bad 
# Using outdated base image with known CVEs
FROM node:14                    # EOL version with known vulns
COPY package.json .
RUN npm install                 # No audit, no lockfile verification
COPY . .
CMD ["node", "server.js"]

安全なコード例

Dockerfile ✅ Good 
# Use current, slim base image with vulnerability scanning
FROM node:22-slim AS build
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --only=production    # Reproducible install from lockfile
RUN npm audit --audit-level=high # Fail on high+ vulns

FROM gcr.io/distroless/nodejs22-debian12
COPY --from=build /app /app
CMD ["/app/server.js"]

対策チェックリスト

  • CI/CDと実行時にコンテナイメージのCVEを継続的にスキャンする
  • 攻撃対象領域を減らすために最小限のベースイメージ（distroless、Alpine）を使用する
  • Dependabot、Renovateなどのツールで依存関係の更新を自動化する
  • SBOMを維持し、すべてのデプロイメント全体でコンポーネントのバージョンを追跡する

8️⃣ CNAS-8 - 不適切な資産管理

Medium

概要

クラウドネイティブ環境では、リソースのスピンアップは簡単ですが、追跡は困難です。孤立したコンテナ、忘れられたネームスペース、古いクラウドリソース、シャドーITデプロイメントは、管理されていない攻撃対象領域を作り出します。適切な資産インベントリがないと、セキュリティチームは存在を知らないものを保護できません。

リスク

忘れられたまたは管理されていないリソースは、古いソフトウェアを実行し、セキュリティ監視が欠けており、古い資格情報を持っていることが多いです。攻撃者は、監視やパッチ適用される可能性が低いため、これらの放置された資産をエントリーポイントとして狙います。

脆弱なコード例

Terraform ❌ Bad 
# Resources created without tagging or lifecycle management
resource "aws_instance" "test_server" {
  ami           = "ami-0abcdef1234567890"
  instance_type = "t3.medium"
  # No tags — who owns this? What's it for?
  # No lifecycle policy — runs forever
}

resource "aws_s3_bucket" "temp_data" {
  bucket = "temp-data-2024"
  # No expiration, no access logging
}

安全なコード例

Terraform ✅ Good 
resource "aws_instance" "test_server" {
  ami           = "ami-0abcdef1234567890"
  instance_type = "t3.medium"

  tags = {
    Name        = "test-server"
    Owner       = "platform-team"
    Environment = "staging"
    ManagedBy   = "terraform"
    ExpiresAt   = "2026-04-30"
  }
}

resource "aws_s3_bucket" "temp_data" {
  bucket = "temp-data-2024"

  tags = {
    Owner     = "data-team"
    ManagedBy = "terraform"
  }

  lifecycle_rule {
    enabled = true
    expiration { days = 90 }
  }
}

対策チェックリスト

  • すべてのクラウドリソース、クラスタ、ネームスペースのリアルタイムインベントリを維持する
  • すべてのリソースに対して必須のタグ付けポリシー（所有者、環境、有効期限）を強制する
  • 孤立したリソースの検出とクリーンアップを自動化する（AWS Config、GCP Asset Inventory）
  • 追跡されない手動のリソース作成を防ぐためにInfrastructure as Codeを実装する

9️⃣ CNAS-9 - 不十分なコンピュートリソースクォータ制限

Medium

概要

適切なリソースクォータと制限がないと、単一の誤動作または侵害されたワークロードが、利用可能なすべてのCPU、メモリ、ストレージを消費し、他のアプリケーションにサービス拒否を引き起こす可能性があります。リソース制限のない環境では、特にクリプトジャッキング攻撃が一般的です。

リスク

攻撃者は、すべてのクラスタリソースを消費する暗号通貨マイニングコンテナやリソース集約的なワークロードをデプロイできます。制限がないと、フォーク爆弾やメモリリークが一つのポッドでノード全体をクラッシュさせ、共存するすべてのワークロードに影響を与える可能性があります。

脆弱なコード例

YAML (Kubernetes) ❌ Bad 
# Pod with no resource limits
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    image: myapp:latest
    # No resources block — can consume unlimited CPU/memory!

安全なコード例

YAML (Kubernetes) ✅ Good 
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    image: myapp:1.2.3
    resources:
      requests:
        cpu: "100m"
        memory: "128Mi"
      limits:
        cpu: "500m"
        memory: "512Mi"
---
# Namespace-level quota
apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-quota
  namespace: production
spec:
  hard:
    requests.cpu: "10"
    requests.memory: "20Gi"
    limits.cpu: "20"
    limits.memory: "40Gi"

対策チェックリスト

  • すべてのコンテナにCPUとメモリのリクエスト/制限を設定する
  • ネームスペースレベルでResourceQuotasとLimitRangesを強制する
  • リソース使用状況を監視し、異常な消費パターンにアラートを出す
  • 無制限のスケーリングを防ぐために、最大ノード制限付きのクラスタオートスケーラーを使用する

🔟 CNAS-10 - 効果的でないログ記録と監視

High

概要

クラウドネイティブ環境は、複数のレイヤーにわたってログを生成します：アプリケーション、コンテナランタイム、オーケストレーター、サービスメッシュ、クラウドプラットフォーム。集中ログ記録、これらのレイヤー間の相関、実行時脅威検出がないと、セキュリティインシデントは検出されないか、発見が遅すぎることになります。

リスク

エフェメラルコンテナは終了時にログを失い、フォレンジック証拠を破壊します。Kubernetes監査ログと実行時監視がないと、攻撃者はアラートをトリガーすることなく、バックドアを作成し、権限を昇格させたり、データを流出させることができます。

脆弱なコード例

YAML (Kubernetes) ❌ Bad 
# No audit policy, no log forwarding
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    image: myapp:latest
    # Logs go to stdout only — lost when pod restarts
    # No audit logging configured on the cluster
    # No runtime security monitoring

安全なコード例

YAML (Kubernetes) ✅ Good 
# Kubernetes audit policy for security events
apiVersion: audit.k8s.io/v1
kind: Policy
rules:
- level: RequestResponse
  resources:
  - group: ""
    resources: ["secrets", "configmaps"]
- level: Metadata
  resources:
  - group: "rbac.authorization.k8s.io"
    resources: ["clusterroles", "clusterrolebindings"]
- level: Metadata
  verbs: ["create", "delete", "patch"]
# Deploy Falco for runtime threat detection
# Forward logs to SIEM via Fluent Bit / Fluentd

対策チェックリスト

  • Kubernetes監査ログを有効化し、集中型SIEMに転送する
  • リアルタイム脅威検出のために実行時セキュリティツール（Falco、Sysdig）をデプロイする
  • ポッドのライフサイクルを超えてコンテナログを保持するためにログ集約（Fluent Bit、Loki）を使用する
  • セキュリティ上重要なイベントに対してアラートを作成する：シークレットアクセス、RBAC変更、ポッドへのexec
