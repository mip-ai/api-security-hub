Consignes de sécurité internes des API

Listes de contrôle de sécurité, perspectives de révision du code et modèles de politiques pour les équipes de développement.

📋 Liste de contrôle de pré-développement

Éléments à vérifier avant de développer un nouveau point de terminaison d'API.

  • La spécification API (OpenAPI) a été créée
  • Les méthodes d'authentification et d'autorisation ont été déterminées
  • Les types de paramètres d'entrée et les contraintes ont été définis
  • Les champs à inclure dans les réponses ont été clairement définis
  • Le format de réponse aux erreurs est standardisé
  • Des exigences de limitation des tarifs ont été établies
  • Le traitement des données personnelles et sensibles a été confirmé
  • La méthode d'authentification a été déterminée (OAuth 2.0, JWT, API Key)
  • Les autorisations requises (portées/rôles) ont été définies pour chaque point de terminaison
  • Des contrôles d'autorisation au niveau des ressources ont été conçus (contre-mesure BOLA)
  • Les fonctions d'administration sont clairement séparées
  • Les méthodes d'expiration et d'actualisation des jetons ont été déterminées
  • La communication utilise TLS 1.2 ou supérieur
  • La méthode de cryptage des données stockées a été déterminée
  • Des politiques de masquage et d'anonymisation des informations personnelles ont été déterminées
  • Les informations sensibles (jetons, mots de passe) ne sont pas incluses dans les journaux
  • Les informations de débogage et les traces de pile ne sont pas incluses dans les réponses de l'API

📜 Modèles de politique de sécurité

Modèles de politiques de base dans le développement d’API internes. Personnalisez-le pour l'adapter à votre projet.

1. Politique d'authentification

PolicyModèle
■ Méthodes d'authentification 
- API destinées aux utilisateurs : OAuth 2.0 (code d'autorisation + PKCE) 
- API de serveur à serveur : OAuth 2.0 (Client Credentials) ou mTLS 
- API d'intégration externes : Clé API + signature HMAC 

■ Gestion des jetons 
- Expiration du jeton d'accès : 15 minutes 
- Expiration du jeton d'actualisation : 7 jours (rotation requise) 
- Stockage des jetons : HttpOnly + Secure + SameSite Cookie 

■ Politique de mot de passe 
- Minimum 12 caractères, au moins une majuscule, une minuscule, un chiffre et un symbole 
- Hash avec bcrypt (facteur de coût 12 ou plus) 
- Protection contre les attaques par liste de mots de passe (intégration de l'API Have I Been Pwned)

2. Politique de conception d'API

PolicyModèle
■ Gestion des versions 
- Gestion des versions via chemin URL : /api/v1/resources 
- Période de support minimum de 6 mois pour les anciennes versions 
- Notifier la dépréciation via l'en-tête Deprecation 

■ Limitation de débit (valeurs par défaut) 
- API générales : 100 requêtes/15 min 
- API d'authentification : 5 requêtes/15 min 
- API publiques : 30 requêtes/min 
- Renvoie toujours les en-têtes RateLimit-* 

■ Réponses 
- Type de contenu : application/json (fixé) 
- Réponses aux erreurs au format RFC 7807 (Détails du problème) 
- Ne restituez jamais les traces de pile en production 
- Pagination : 20 éléments par défaut, maximum 100 éléments

3. Politique de journalisation et d'audit

PolicyModèle
■ Champs de journal obligatoires 
- Horodatage (ISO 8601, UTC) 
- ID de demande (UUID pour la traçabilité) 
- ID utilisateur / ID client API 
- Méthode HTTP + point de terminaison 
- Code d'état 
- IP source 
- Temps de réponse 

■ Champs de journal interdits (données sensibles) 
- Mots de passe / jetons / clés API 
- Numéros de carte de crédit 
- Affichage PII complet (masquage requis) 

■ Surveillance des alertes 
- Échecs d'authentification : alerte à 5+/min 
- Erreurs 403 : Alerte à 10+/min 
- 500 erreurs : Alerte immédiate sur 1 occurrence 
- Limite de débit dépassée : analyse de modèle

🚨 Flux de réponse aux incidents

Phase 1 : Détection

Détectez les incidents grâce à des alertes de surveillance, des rapports utilisateur ou des notifications externes. Déterminez la gravité grâce au tri initial.

0-30 minutes

Phase 2 : Confinement

Empêcher l’escalade des dégâts. Invalidez les clés API, suspendez temporairement les points de terminaison concernés et identifiez l'étendue de l'impact.

30 minutes - 2 heures

Phase 3 : Éradication et rétablissement

Corrigez les vulnérabilités, appliquez des correctifs et restaurez les services. Avertissez les utilisateurs concernés.

2-24 heures

Phase 4 : Analyse post-incident

Effectuer une autopsie. Effectuer une analyse des causes profondes, élaborer des mesures de prévention des récidives et documenter les résultats.

1 à 5 jours ouvrables
