OWASP Web Application Top 10 (2025)

Les 10 risques de sécurité les plus critiques pour les applications web et comment les atténuer.

Qu'est-ce que l'OWASP Web Application Top 10 ?

L'OWASP Top 10 est le document de sensibilisation le plus largement reconnu en matière de sécurité des applications web. L'édition 2025 reflète le paysage des menaces le plus récent, introduisant de nouvelles catégories telles que les Défaillances de la Chaîne d'Approvisionnement Logicielle et la Gestion Inadéquate des Conditions Exceptionnelles.

1️⃣ A01 - Contrôle d'Accès Défaillant

Critical

Aperçu

Le contrôle d'accès applique des politiques pour que les utilisateurs ne puissent pas agir en dehors de leurs permissions prévues. Les défaillances conduisent typiquement à la divulgation non autorisée d'informations, la modification ou la destruction de données, ou l'exécution de fonctions en dehors des limites de l'utilisateur.

Risque

Les attaquants peuvent exploiter les failles de contrôle d'accès pour accéder aux comptes d'autres utilisateurs, consulter des fichiers sensibles, modifier les données d'autres utilisateurs ou changer les droits d'accès.

Exemple de Code Vulnérable

Python ❌ Bad 
# User ID taken directly from request without authorization check
@app.route('/api/user/<user_id>/profile')
def get_profile(user_id):
    user = db.get_user(user_id)  # No ownership check!
    return jsonify(user.to_dict())

Exemple de Code Sécurisé

Python ✅ Good 
@app.route('/api/user/<user_id>/profile')
@login_required
def get_profile(user_id):
    # Verify the requesting user owns this resource
    if current_user.id != user_id and not current_user.is_admin:
        abort(403)

    user = db.get_user(user_id)
    if not user:
        abort(404)
    return jsonify(user.to_dict())

Liste de Vérification d'Atténuation

  • Refuser l'accès par défaut sauf pour les ressources publiques
  • Implémenter des vérifications de contrôle d'accès côté serveur à chaque requête
  • Appliquer la propriété des enregistrements au lieu d'accepter les IDs soumis par l'utilisateur
  • Journaliser les échecs de contrôle d'accès et alerter les administrateurs sur les violations répétées

2️⃣ A02 - Mauvaise Configuration de Sécurité

High

Aperçu

La mauvaise configuration de sécurité se produit lorsque les paramètres de sécurité sont définis, implémentés ou maintenus incorrectement. Cela inclut le manque de renforcement de sécurité, les fonctionnalités inutiles activées, les comptes par défaut avec des mots de passe inchangés, les messages d'erreur trop détaillés et les en-têtes de sécurité HTTP mal configurés.

Risque

Les serveurs, frameworks ou services cloud mal configurés peuvent exposer des données sensibles, permettre un accès non autorisé ou fournir aux attaquants des informations pour planifier des attaques supplémentaires.

Exemple de Code Vulnérable

Python ❌ Bad 
# Debug mode enabled in production, verbose errors exposed
app = Flask(__name__)
app.config['DEBUG'] = True  # Exposes stack traces!
app.config['SECRET_KEY'] = 'default-secret'  # Default key!

Exemple de Code Sécurisé

Python ✅ Good 
import os

app = Flask(__name__)
app.config['DEBUG'] = False
app.config['SECRET_KEY'] = os.environ['SECRET_KEY']

@app.after_request
def set_security_headers(response):
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['X-Frame-Options'] = 'DENY'
    response.headers['Strict-Transport-Security'] = 'max-age=31536000; includeSubDomains'
    response.headers['Content-Security-Policy'] = "default-src 'self'"
    return response

Liste de Vérification d'Atténuation

  • Implémenter un processus de renforcement reproductible pour tous les environnements
  • Supprimer ou désactiver les fonctionnalités, frameworks et comptes par défaut inutiles
  • Configurer les en-têtes de sécurité appropriés (CSP, HSTS, X-Frame-Options)
  • Utiliser des outils automatisés pour vérifier l'efficacité des configurations

3️⃣ A03 - Défaillances de la Chaîne d'Approvisionnement Logicielle

High

Aperçu

Nouveau en 2025. Se concentre sur les risques liés aux composants tiers, aux dépendances et aux pipelines CI/CD. Les attaquants ciblent la chaîne d'approvisionnement logicielle en compromettant des packages, en injectant du code malveillant dans des bibliothèques open source ou en exploitant les vulnérabilités du pipeline de build.

Risque

Une seule dépendance compromise peut affecter des milliers d'applications. Les attaques de la chaîne d'approvisionnement peuvent conduire au vol de données, à l'installation de portes dérobées ou à la compromission complète du système avec une détection minimale.

Exemple de Code Vulnérable

JavaScript ❌ Bad 
// package.json with unpinned dependencies
{
  "dependencies": {
    "express": "*",           // Any version!
    "lodash": "^4.0.0",       // Wide range
    "unknown-pkg": "^1.0.0"   // Unvetted package
  }
}

Exemple de Code Sécurisé

JavaScript ✅ Good 
// package.json with pinned versions + lockfile + audit
{
  "dependencies": {
    "express": "4.21.2",
    "lodash": "4.17.21"
  },
  "scripts": {
    "preinstall": "npx npm-audit-resolver",
    "integrity-check": "npm audit signatures"
  }
}
// Also: use package-lock.json, enable Dependabot/Renovate,
// generate SBOM, verify package provenance

Liste de Vérification d'Atténuation

  • Fixer les versions des dépendances et utiliser des fichiers de verrouillage pour des builds reproductibles
  • Activer l'analyse automatisée des dépendances (Dependabot, Snyk, Renovate)
  • Générer et maintenir une Nomenclature Logicielle (SBOM)
  • Vérifier la provenance et les signatures des packages avant inclusion

4️⃣ A04 - Défaillances Cryptographiques

High

Aperçu

Défaillances liées à la cryptographie qui conduisent souvent à l'exposition de données sensibles. Cela inclut l'utilisation d'algorithmes obsolètes, la génération faible de clés, le manque de chiffrement pour les données en transit ou au repos, et la validation incorrecte des certificats.

Risque

Un chiffrement faible ou manquant peut exposer des mots de passe, des numéros de carte de crédit, des dossiers médicaux, des informations personnelles et des secrets commerciaux, pouvant entraîner des violations réglementaires (RGPD, PCI DSS).

Exemple de Code Vulnérable

Python ❌ Bad 
import hashlib

# Storing passwords with weak hashing
def store_password(password: str) -> str:
    return hashlib.md5(password.encode()).hexdigest()  # MD5 is broken!

Exemple de Code Sécurisé

Python ✅ Good 
import bcrypt

def store_password(password: str) -> bytes:
    # Use bcrypt with automatic salting
    salt = bcrypt.gensalt(rounds=12)
    return bcrypt.hashpw(password.encode(), salt)

def verify_password(password: str, hashed: bytes) -> bool:
    return bcrypt.checkpw(password.encode(), hashed)

Liste de Vérification d'Atténuation

  • Utiliser des algorithmes forts et actuels (AES-256, bcrypt/argon2, SHA-256+)
  • Chiffrer toutes les données sensibles en transit (TLS 1.2+) et au repos
  • Ne jamais utiliser d'algorithmes obsolètes (MD5, SHA-1, DES, RC4)
  • Implémenter une gestion appropriée des clés avec des politiques de rotation

5️⃣ A05 - Injection

Critical

Aperçu

Les failles d'injection se produisent lorsque des données non fiables sont envoyées à un interpréteur dans le cadre d'une commande ou d'une requête. L'injection SQL, l'injection NoSQL, l'injection de commandes OS et le Cross-Site Scripting (XSS) sont les formes les plus courantes. Des données hostiles peuvent tromper l'interpréteur pour exécuter des commandes non prévues.

Risque

L'injection peut entraîner la perte de données, la corruption, l'accès non autorisé, la prise de contrôle complète de l'hôte ou le déni de service. L'injection SQL seule reste l'un des vecteurs d'attaque les plus dangereux et les plus répandus.

Exemple de Code Vulnérable

Python ❌ Bad 
# SQL injection via string concatenation
def get_user(username: str):
    query = f"SELECT * FROM users WHERE name = '{username}'"
    return db.execute(query)  # username = "' OR '1'='1"

Exemple de Code Sécurisé

Python ✅ Good 
# Parameterized queries prevent SQL injection
def get_user(username: str):
    query = "SELECT * FROM users WHERE name = %s"
    return db.execute(query, (username,))

# For XSS prevention, escape output
from markupsafe import escape

def render_comment(comment: str) -> str:
    return f"<p>{escape(comment)}</p>"

Liste de Vérification d'Atténuation

  • Utiliser des requêtes paramétrées ou des instructions préparées pour toutes les opérations de base de données
  • Utiliser un encodage de sortie contextuel pour prévenir le XSS
  • Valider et assainir toutes les entrées utilisateur côté serveur
  • Utiliser des ORM avec des requêtes paramétrées plutôt que la construction de chaînes SQL brutes

6️⃣ A06 - Conception Non Sécurisée

High

Aperçu

La conception non sécurisée fait référence aux risques liés aux défauts de conception et d'architecture. Elle nécessite l'utilisation de la modélisation des menaces, de modèles de conception sécurisés et d'architectures de référence. Une conception non sécurisée ne peut pas être corrigée par une implémentation parfaite ; les contrôles de sécurité nécessaires n'ont jamais été créés pour se défendre contre des attaques spécifiques.

Risque

Les défauts de conception peuvent conduire à des vulnérabilités de logique métier difficiles à détecter avec des outils automatisés. L'absence de limites de débit sur les opérations sensibles, le manque d'authentification multifacteur pour les actions critiques ou la détection insuffisante de la fraude sont des défaillances au niveau de la conception.

Exemple de Code Vulnérable

Python ❌ Bad 
# Password reset with no rate limit or verification
@app.route('/reset-password', methods=['POST'])
def reset_password():
    email = request.form['email']
    new_pass = request.form['new_password']
    user = db.find_by_email(email)
    user.password = hash_password(new_pass)  # No token verification!
    db.save(user)

Exemple de Code Sécurisé

Python ✅ Good 
from datetime import datetime, timedelta

@app.route('/reset-password', methods=['POST'])
@rate_limit("3/hour")
def reset_password():
    token = request.form['token']
    new_pass = request.form['new_password']

    # Verify time-limited, single-use token
    reset_req = db.find_reset_token(token)
    if not reset_req or reset_req.used or reset_req.expires < datetime.utcnow():
        abort(400, "Invalid or expired token")

    # Enforce password complexity
    if not meets_password_policy(new_pass):
        abort(400, "Password does not meet requirements")

    reset_req.user.password = hash_password(new_pass)
    reset_req.used = True
    db.save_all([reset_req.user, reset_req])

Liste de Vérification d'Atténuation

  • Intégrer la modélisation des menaces dans le cycle de vie du développement
  • Appliquer la limitation de débit et les contrôles anti-automatisation sur les opérations sensibles
  • Utiliser des modèles de conception sécurisés et des architectures de référence
  • Écrire des tests unitaires et d'intégration pour les cas d'abus, pas seulement les cas d'utilisation

7️⃣ A07 - Défaillances d'Authentification

High

Aperçu

La confirmation de l'identité de l'utilisateur, l'authentification et la gestion des sessions sont critiques. Les défaillances d'authentification se produisent lorsque les applications permettent le credential stuffing, la force brute, les mots de passe faibles, ou ont une gestion de session défectueuse comme des IDs de session qui ne tournent pas après la connexion.

Risque

Les attaquants peuvent accéder aux comptes utilisateur via le credential stuffing automatisé, la force brute ou le détournement de session. Les comptes compromis peuvent conduire au vol d'identité, à la fraude et aux violations de données.

Exemple de Code Vulnérable

Python ❌ Bad 
# No brute force protection, weak session handling
@app.route('/login', methods=['POST'])
def login():
    user = db.find_by_email(request.form['email'])
    if user and user.password == request.form['password']:  # Plain comparison!
        session['user'] = user.id  # Session ID not rotated
        return redirect('/dashboard')

Exemple de Code Sécurisé

Python ✅ Good 
from flask_limiter import Limiter

limiter = Limiter(app, default_limits=["100/hour"])

@app.route('/login', methods=['POST'])
@limiter.limit("5/minute")
def login():
    user = db.find_by_email(request.form['email'])
    if not user or not bcrypt.checkpw(
        request.form['password'].encode(), user.password_hash
    ):
        # Generic error to prevent user enumeration
        return "Invalid credentials", 401

    # Rotate session ID after authentication
    session.regenerate()
    session['user'] = user.id

    # Check for MFA requirement
    if user.mfa_enabled:
        return redirect('/mfa-verify')
    return redirect('/dashboard')

Liste de Vérification d'Atténuation

  • Implémenter l'authentification multifacteur (MFA) pour toutes les opérations critiques
  • Appliquer une protection contre la force brute avec limitation de débit et verrouillage de compte
  • Faire tourner les IDs de session après l'authentification et utiliser des flags de session sécurisés
  • Utiliser des messages d'erreur génériques pour empêcher l'énumération des utilisateurs

8️⃣ A08 - Défaillances d'Intégrité des Logiciels ou Données

High

Aperçu

Les défaillances d'intégrité des logiciels et des données se rapportent au code et à l'infrastructure qui ne protègent pas contre les violations d'intégrité. Cela inclut la désérialisation non sécurisée, l'utilisation de CDN ou plugins non fiables sans vérification d'intégrité, et les mécanismes de mise à jour automatique sans mises à jour signées.

Risque

La désérialisation non sécurisée peut conduire à l'exécution de code à distance. Le chargement de scripts depuis des CDN non fiables sans Subresource Integrity (SRI) peut permettre aux attaquants d'injecter du code malveillant dans votre application.

Exemple de Code Vulnérable

HTML ❌ Bad 
<!-- Loading external scripts without integrity checks -->
<script src="https://cdn.example.com/lib.js"></script>

<!-- Insecure deserialization in Python -->
import pickle
data = pickle.loads(user_input)  # Arbitrary code execution!

Exemple de Code Sécurisé

HTML ✅ Good 
<!-- Use Subresource Integrity (SRI) for external scripts -->
<script src="https://cdn.example.com/lib.js"
  integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K..."
  crossorigin="anonymous"></script>

# Use safe deserialization in Python
import json
data = json.loads(user_input)  # Safe: only parses JSON data

Liste de Vérification d'Atténuation

  • Utiliser Subresource Integrity (SRI) pour tous les scripts et feuilles de style externes
  • Ne jamais utiliser de désérialisation non sécurisée (pickle, yaml.load) sur des entrées non fiables
  • Vérifier les signatures numériques sur les mises à jour logicielles et les artefacts CI/CD
  • Implémenter des vérifications d'intégrité dans les pipelines CI/CD pour prévenir la falsification

9️⃣ A09 - Défaillances de Journalisation et d'Alerte de Sécurité

Medium

Aperçu

Sans journalisation, surveillance et alertes suffisantes, les violations ne peuvent pas être détectées en temps opportun. Une journalisation insuffisante, une intégration inefficace avec les systèmes de réponse aux incidents et le manque d'alertes en temps réel permettent aux attaquants de continuer à attaquer les systèmes, de maintenir la persistance et de falsifier ou extraire des données.

Risque

Sans journalisation appropriée, les attaquants peuvent opérer sans être détectés pendant de longues périodes. La plupart des études sur les violations montrent que le temps moyen de détection d'une violation dépasse 200 jours, souvent découverte par des parties externes plutôt que par la surveillance interne.

Exemple de Code Vulnérable

Python ❌ Bad 
# No logging of security-relevant events
@app.route('/login', methods=['POST'])
def login():
    user = authenticate(request.form)
    if not user:
        return "Login failed", 401  # No record of failure
    return redirect('/dashboard')

Exemple de Code Sécurisé

Python ✅ Good 
import logging
from datetime import datetime

security_log = logging.getLogger('security')

@app.route('/login', methods=['POST'])
def login():
    user = authenticate(request.form)
    if not user:
        security_log.warning(
            "LOGIN_FAILED | ip=%s | email=%s | time=%s",
            request.remote_addr,
            request.form.get('email', 'unknown'),
            datetime.utcnow().isoformat(),
        )
        check_brute_force(request.remote_addr)
        return "Invalid credentials", 401

    security_log.info(
        "LOGIN_SUCCESS | user_id=%s | ip=%s",
        user.id, request.remote_addr,
    )
    return redirect('/dashboard')

Liste de Vérification d'Atténuation

  • Journaliser toutes les tentatives d'authentification, les échecs de contrôle d'accès et les erreurs de validation d'entrée
  • Utiliser un système centralisé de gestion des journaux avec protection contre la falsification
  • Implémenter des alertes en temps réel pour les modèles d'activité suspecte
  • Établir un plan de réponse aux incidents avec des procédures d'escalade définies

🔟 A10 - Gestion Inadéquate des Conditions Exceptionnelles

Medium

Aperçu

Nouveau en 2025. Les applications qui gèrent incorrectement les erreurs, exceptions et cas limites peuvent exposer des informations sensibles, entrer dans des états incohérents ou créer des conditions exploitables. Cela inclut les messages d'erreur détaillés en production, les exceptions non capturées qui contournent les contrôles de sécurité et les conditions de course.

Risque

La gestion inadéquate des erreurs peut exposer des traces de pile, des détails de base de données ou des chemins internes. Les conditions de course dans les vérifications de sécurité peuvent permettre des attaques TOCTOU (Time-of-Check-to-Time-of-Use), contournant l'autorisation ou la validation de paiement.

Exemple de Code Vulnérable

Python ❌ Bad 
# Exposing internal details in error messages
@app.route('/api/data')
def get_data():
    try:
        result = db.query(request.args['q'])
        return jsonify(result)
    except Exception as e:
        return jsonify({"error": str(e)}), 500  # Leaks DB details!

Exemple de Code Sécurisé

Python ✅ Good 
import uuid, logging

logger = logging.getLogger(__name__)

@app.errorhandler(Exception)
def handle_exception(e):
    error_id = str(uuid.uuid4())
    logger.error("Unhandled exception [%s]: %s", error_id, e, exc_info=True)

    # Return generic error with reference ID for support
    return jsonify({
        "error": "An internal error occurred",
        "reference": error_id,
    }), 500

@app.route('/api/data')
def get_data():
    q = request.args.get('q')
    if not q or not is_valid_query(q):
        return jsonify({"error": "Invalid query parameter"}), 400

    result = db.query(q)
    return jsonify(result)

Liste de Vérification d'Atténuation

  • Ne jamais exposer de traces de pile, d'erreurs de base de données ou de chemins internes en production
  • Implémenter un gestionnaire d'erreurs global qui renvoie des messages génériques avec des IDs de référence
  • Tester les conditions de course dans les opérations critiques de sécurité
  • S'assurer que tous les chemins d'erreur maintiennent les invariants de sécurité et nettoient les ressources
