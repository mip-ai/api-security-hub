Meilleures pratiques d’authentification et d’autorisation

Un guide pratique sur l'authentification et l'autorisation des API, y compris OAuth 2.0, JWT et la gestion des clés API.

🔑 OAuth 2.0

OAuth 2.0 est le cadre d'autorisation le plus largement utilisé pour l'accès aux API. La sélection du type de subvention approprié est essentielle.

Choisir un type de subvention

Type de subvention Cas d'utilisation Niveau de sécurité
Authorization Code + PKCE SPAs et applications mobiles (recommandés) Le plus haut
Authorization Code Applications Web côté serveur Haut
Client Credentials Communication de serveur à serveur (M2M) Moyen
Implicite (obsolète) SPA hérités Faible
Mot de passe du propriétaire de la ressource (obsolète) Première partie hautement fiable uniquement Faible
N'utilisez pas de subvention implicite

L'octroi implicite est désormais obsolète car le jeton est exposé dans le fragment d'URL. Utilisez le code d'autorisation + PKCE pour les SPA.

Code d'autorisation + flux PKCE

JavaScript Génération de défis PKCE 
// 1. Générer aléatoirement code_verifier
function generateCodeVerifier() {
  const array = new Uint8Array(32);
  crypto.getRandomValues(array);
  return btoa(String.fromCharCode(...array))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}

// 2. Générez code_challenge avec SHA-256
async function generateCodeChallenge(verifier) {
  const encoder = new TextEncoder();
  const data = encoder.encode(verifier);
  const digest = await crypto.subtle.digest('SHA-256', data);
  return btoa(String.fromCharCode(...new Uint8Array(digest)))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}

// 3. Demande d'autorisation
const verifier = generateCodeVerifier();
const challenge = await generateCodeChallenge(verifier);

const authUrl = `https://auth.example.com/authorize?` +
  `response_type=code&` +
  `client_id=${clientId}&` +
  `redirect_uri=${redirectUri}&` +
  `code_challenge=${challenge}&` +
  `code_challenge_method=S256&` +
  `scope=read write`;

🎫 JWT (jeton Web JSON)

JWT est largement utilisé comme jeton d'authentification sans état, mais les erreurs de mise en œuvre conduisent directement à des vulnérabilités de sécurité.

Considérations sur la sécurité JWT

✅Faire

  • Spécifier explicitement l'algorithme (HS256 / RS256)
  • Définir un délai d'expiration (exp) court (15 minutes ou moins)
  • Valider l'émetteur (iss) et l'audience (aud)
  • Générez des clés secrètes d'une longueur suffisante (256 bits ou plus)
  • Implémenter la rotation pour les jetons d'actualisation

❌ Ne le fais pas

  • Stockez les informations sensibles dans la charge utile JWT (Base64 n'est pas un cryptage)
  • Autoriser alg : "aucun"
  • Stocker les jetons dans localStorage (vulnérable à XSS)
  • Négliger la gestion de la révocation des jetons
  • Utiliser une clé publique RS256 comme clé secrète HS256

Implémentation sécurisée de JWT

JavaScript (Node.js) Émission et vérification des jetons 
const jwt = require('jsonwebtoken');

// --- Émission de jetons ---
function issueTokens(user) {
  const accessToken = jwt.sign(
    { sub: user.id, role: user.role },
    process.env.JWT_SECRET,
    {
      algorithm: 'HS256',
      expiresIn: '15m',
      issuer: 'api.example.com',
      audience: 'app.example.com',
    }
  );

  const refreshToken = jwt.sign(
    { sub: user.id, type: 'refresh' },
    process.env.REFRESH_SECRET,
    { algorithm: 'HS256', expiresIn: '7d' }
  );

  return { accessToken, refreshToken };
}

// --- Vérification du jeton ---
function verifyAccessToken(token) {
  return jwt.verify(token, process.env.JWT_SECRET, {
    algorithms: ['HS256'],   // ← Précisez toujours !
    issuer: 'api.example.com',
    audience: 'app.example.com',
  });
}

Comparaison des emplacements de stockage des jetons

Emplacement de stockage Résistance XSS Résistance CSRF Recommandation
HttpOnly Cookie ◎ △ (nécessite le paramètre SameSite) Recommandée
Mémoire (variable) ◎ ◎ Perdu dans la navigation des pages
localStorage ✕ (peut être volé via XSS) ◎ Non recommandé
sessionStorage ✕ (peut être volé via XSS) ◎ Utilisation limitée
Recommandé : HttpOnly + Secure + Cookie SameSite

Stocker le jeton d'accès dans un cookie avec les attributs HttpOnly, Secure et SameSite=Strict est l'approche la plus sûre. Cela empêche le vol de jetons via des attaques XSS.

🗝 Gestion des clés API

Les clés API constituent un mécanisme d’authentification simple, mais une mauvaise gestion peut entraîner de graves risques.

Meilleures pratiques en matière de clés API

Génération et stockage

Générez avec une entropie suffisante (256 bits ou plus). Stocker haché ; ne conservez jamais de texte brut dans la base de données.

Restriction de portée

Attribuez des étendues minimales par clé (lecture seule, ressources spécifiques uniquement, etc.). Évitez les autorisations génériques.

Rotation

Faites pivoter les clés régulièrement (dans les 90 jours). Définissez un délai de grâce pour les anciennes clés. Révoquer immédiatement en cas de fuite.

Méthode de transmission

Envoyer via l'en-tête (X-API-Key ou autorisation). N'incluez pas les paramètres de requête d'URL (ils apparaissent dans les journaux).

Gestion des variables d'environnement

Ne codez pas en dur les clés dans le code source. Gérez-les avec des variables d'environnement ou un gestionnaire de secrets.

Surveillance et journalisation

Surveillez l’utilisation des clés API. Détectez les modèles anormaux (demandes massives, accès à partir d’adresses IP inconnues).

Exemple de mise en œuvre de la validation de clé API

JavaScript (Express) Validation de la clé API hachée 
const crypto = require('crypto');

// Hachez la clé API avec SHA-256 et comparez
async function validateApiKey(req, res, next) {
  const apiKey = req.headers['x-api-key'];
  if (!apiKey) {
    return res.status(401).json({ error: 'API key required' });
  }

  // Hachez et recherchez dans la base de données (timingSafeEqual n'est pas nécessaire puisque nous comparons les hachages)
  const hashedKey = crypto
    .createHash('sha256')
    .update(apiKey)
    .digest('hex');

  const keyRecord = await db.findApiKey(hashedKey);

  if (!keyRecord || keyRecord.revokedAt) {
    return res.status(403).json({ error: 'Invalid API key' });
  }

  // Vérifier l'expiration
  if (keyRecord.expiresAt && new Date() > keyRecord.expiresAt) {
    return res.status(403).json({ error: 'API key expired' });
  }

  req.apiClient = keyRecord;
  next();
}
