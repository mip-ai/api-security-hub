🔥 Incidents majeurs

Critical 29 janvier 2025 Exposition des données

DeepSeek expose plus d'un million de journaux via une base de données non authentifiée - Fuite de l'historique des discussions et des clés API

Wiz Research a découvert que la base de données ClickHouse de DeepSeek était accessible publiquement sans authentification sur deux sous-domaines. Plus d'un million d'entrées de journal contenaient l'historique des discussions en texte brut, les clés secrètes de l'API et les métadonnées opérationnelles du backend. Les attaquants pourraient voler des mots de passe et des fichiers locaux directement depuis le serveur. Suite à une divulgation responsable, DeepSeek a immédiatement sécurisé la base de données. Une attaque DDoS à grande échelle contre l'API et l'interface de chat Web s'est également produite le 27 janvier.

Source : Blog Wiz

Critical 10 mai 2024 Violation de données

49 millions de dossiers clients volés via l'abus de l'API du portail des partenaires Dell

Un attaquant a enregistré un faux compte partenaire sur le portail partenaire de Dell et y a obtenu l'accès dans les 48 heures. L'API du portail n'avait aucun contrôle d'autorisation ni limitation de débit. Pendant trois semaines, l'attaquant a envoyé environ 5 000 requêtes par minute, volant 49 millions d'enregistrements contenant des noms, des adresses, des numéros de service et des informations sur les commandes. Les données ont été vendues sur un forum de hacking.

Source : BleepingComputer

High 16 janvier 2024 Violation de données

Fuite des données de 15 millions d'utilisateurs de Trello via le scraping de l'API

Un attaquant a exploité l'API REST non protégée de Trello, qui permettait des requêtes non authentifiées par adresse e-mail. En soumettant des adresses e-mail provenant de bases de données précédemment violées, l'attaquant a récupéré plus de 15 millions de profils d'utilisateurs (e-mails, noms complets, noms d'utilisateur, informations sur le forum) et les a publiés sur un forum de piratage. Atlassian a ensuite restreint le point de terminaison.

Source : BleepingComputer

High juillet 2025 Fuite de clé API

Un employé de DOGE publie la clé API privée xAI sur GitHub - Accès à plus de 52 LLM internes exposés

Un employé du Département américain de l'efficacité gouvernementale (DOGE) a validé un script (agent.py) contenant la clé API privée de xAI dans un référentiel GitHub public. La clé offrait un accès illimité à plus de 52 modèles LLM, dont « grok 4-0709 », créé quatre jours plus tôt. GitGuardian a détecté la fuite, mais la clé n'a pas été immédiatement révoquée. Une fuite similaire s’était également produite en mai 2025.

Source : Krebs sur la sécurité

🛡 CVE notables

Active Exploitation Mars 2025 (exploitation active observée) SSRF

CVE-2024-27564 - Plus de 10 000 tentatives d'attaque sur la vulnérabilité ChatGPT SSRF

Une vulnérabilité SSRF (Server-Side Request Forgery) dans le composant pictureproxy.php de ChatGPT. Un attaquant non authentifié pourrait injecter des URL contrefaites pour forcer le serveur à exécuter des requêtes internes non autorisées. Plus de 10 000 tentatives d’attaque ont été enregistrées à partir d’une seule adresse IP, 33 % des cibles étant des organisations américaines (secteurs financier, gouvernemental et de la santé). La vulnérabilité est particulièrement dangereuse car elle peut être exploitée à grande échelle sans authentification.

Source : SecurityWeek

Leçon apprise : SSRF constitue une menace sérieuse pour les API

CVE-2024-27564 est un exemple classique d'OWASP API7 (SSRF). Les API qui acceptent la saisie d'URL provenant de sources externes nécessitent une validation stricte pour empêcher l'accès aux réseaux internes. Voir le Top 10 OWASP → API7.

📊 Rapports et tendances du secteur

Les CVE liés à l'IA augmentent de 1 025 % d'une année sur l'autre

Selon le rapport API ThreatStats 2025 de Wallarm, 439 CVE liés à l'IA ont été enregistrés en 2024, soit une augmentation de 1 025 % d'une année sur l'autre. Plus de 50 % des entrées du catalogue de vulnérabilités exploitées connues (KEV) de CISA sont liées aux API. Les échecs de contrôle d'accès ont augmenté de 40 % au total, et les échecs d'autorisation critiques ont augmenté de 36 %.

Wallarm 2025

99 % des organisations ont rencontré des problèmes de sécurité des API

Selon le rapport du premier trimestre 2025 de Salt Security, 99 % des organisations interrogées ont rencontré au moins un problème de sécurité des API au cours des 12 derniers mois. Les principaux risques étaient les attaques par injection et le BOLA (Broken Object Level Authorization), représentant plus d'un tiers de tous les incidents. 95 % des attaques API provenaient de sessions authentifiées.

Salt Security Q1 2025

30 000 espaces de travail Postman découverts exposant des clés API

Les chercheurs en sécurité ont découvert 30 000 espaces de travail Postman exposés publiquement sans contrôles de sécurité. Ils contenaient des clés API en direct, des jetons d'accès et des charges utiles sensibles, dont beaucoup fournissaient un accès direct aux environnements de production. Ce cas met en évidence le risque d’informations d’identification « shadow API » dans les outils de collaboration des développeurs.

Wallarm 2025

📈 Tendances de sécurité des API 2024-2025

1. Augmentation rapide des vulnérabilités des API AI/LLM

Les vulnérabilités dans les API basées sur l'IA telles que ChatGPT et DeepSeek sont en augmentation. L’injection rapide, le SSRF et l’exposition des données sont les principales menaces. Les mesures de sécurité ne suivent pas l’adoption rapide de l’IA.

2. L'autorisation brisée reste la principale menace

BOLA (Broken Object Level Authorization) continue d’être la vulnérabilité la plus répandue. As seen in the Dell and Trello cases, the lack of basic authorization checks leads to large-scale data breaches.

3. Attaques provenant de sessions authentifiées

95 % des attaques se produisent après l'authentification. Les attaquants disposant de jetons d’accès légitimes exploitent les failles d’autorisation pour se déplacer latéralement, ce qui en fait la technique d’attaque dominante.

4. Fuites d'informations d'identification via les outils de développement

Les fuites de clés API provenant d'outils de développement tels que GitHub et Postman sont devenues un problème critique. L'adoption d'outils de détection automatisés (par exemple, GitGuardian) et de pratiques rigoureuses de gestion des secrets sont essentielles.

✅ Actions à entreprendre maintenant

Contre-mesures tirées de ces incidents
  • Appliquer les contrôles d'autorisation – Implémenter une autorisation au niveau de l'objet sur tous les points de terminaison de l'API (leçon de Dell/Trello)
  • Mettre en œuvre une limitation du débit : définissez des limites de débit appropriées par point de terminaison (Dell n'avait aucune limite, autorisant 5 000 requêtes par minute)
  • Automatiser l'analyse des secrets : Détectez automatiquement les fuites d'informations d'identification dans les référentiels/espaces de travail publics sur GitHub, Postman, etc.
  • Mettre en œuvre des contre-mesures SSRF – Valider les URL fournies par l'utilisateur et bloquer les plages d'adresses IP privées
  • Auditer les paramètres d'exposition de la base de données - Vérifier régulièrement que les interfaces de gestion ne sont pas exposées involontairement (leçon de DeepSeek)
  • Évaluation de la sécurité des points de terminaison AI/LLM – Effectuer des évaluations des risques en cas d'injection rapide et de fuite de données.
  • Vérifications de vulnérabilité des dépendances – Intégrez npm audit / Snyk dans CI/CD pour détecter automatiquement les CVE connus
