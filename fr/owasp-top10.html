Top 10 de la sécurité des API OWASP (2023)

Explique les 10 risques de sécurité les plus critiques spécifiques aux API et comment les atténuer.

Qu'est-ce que le Top 10 de la sécurité des API OWASP ?

Un classement des risques de sécurité spécifiques aux API publié par l'OWASP (Open Worldwide Application Security Project). L'édition 2023 fournit la dernière liste reflétant les menaces qui ont augmenté avec l'adoption généralisée des API.

1️⃣ API1 : 2023 – Autorisation au niveau des objets brisés (BOLA)

Critical

Aperçu

Une vulnérabilité qui permet un accès non autorisé aux données d'autres utilisateurs en manipulant les identifiants d'objet. Il s’agit de la vulnérabilité la plus fréquente et la plus facile à exploiter.

Risque

Les attaquants peuvent accéder aux informations personnelles d'autres utilisateurs, aux détails de paiement, à l'historique des commandes et bien plus encore en modifiant simplement l'ID dans l'URL ou les paramètres de la demande.

Exemple de code vulnérable

JavaScript (Express) ❌ Bad 
// Reçoit l'ID et renvoie les données directement - pas de contrôle d'autorisation
app.get('/api/users/:id/orders', async (req, res) => {
  const orders = await Order.find({ userId: req.params.id });
  res.json(orders);
});

Exemple de code sécurisé

JavaScript (Express) ✅ Good 
// Vérification par rapport à l'ID de l'utilisateur authentifié pour le contrôle d'autorisation
app.get('/api/users/:id/orders', authenticate, async (req, res) => {
  if (req.user.id !== req.params.id && !req.user.isAdmin) {
    return res.status(403).json({ error: 'Forbidden' });
  }
  const orders = await Order.find({ userId: req.params.id });
  res.json(orders);
});

Liste de contrôle d'atténuation

  • Implémenter des contrôles d'autorisation au niveau des objets sur tous les points de terminaison
  • Utilisez des UUID au lieu d'ID séquentiels prévisibles
  • Centraliser la logique d'autorisation dans le middleware
  • Détectez les contournements d’autorisation avec des tests automatisés

2️⃣ API2:2023 - Authentification rompue

Critical

Aperçu

Une vulnérabilité qui permet aux attaquants de s'emparer de comptes d'utilisateurs légitimes en raison de la mise en œuvre défectueuse du mécanisme d'authentification.

Problèmes courants

Politiques de mot de passe faibles, gestion inappropriée des jetons, manque de protection contre la force brute, gestion de session inadéquate, etc.

JavaScript ✅ Exemple d'atténuation : vérification JWT 
const jwt = require('jsonwebtoken');

function authenticate(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'Token required' });

  try {
    // Spécifie explicitement l'algorithme (empêche l'attaque alg: none)
    const decoded = jwt.verify(token, process.env.JWT_SECRET, {
      algorithms: ['HS256'],
      issuer: 'your-app',
    });
    req.user = decoded;
    next();
  } catch (err) {
    res.status(401).json({ error: 'Invalid token' });
  }
}
  • Spécifiez explicitement l'algorithme lors de la vérification des signatures JWT
  • Définissez des délais d'expiration courts pour les jetons (jetons d'accès : dans les 15 minutes)
  • Implémenter la rotation des jetons d'actualisation
  • Appliquer la limitation du taux de tentatives de connexion

3️⃣ API3:2023 - Autorisation au niveau de la propriété d'un objet cassé

High

Aperçu

Une vulnérabilité où le contrôle d'accès aux propriétés (champs) des objets inclus dans les réponses API est insuffisant. Cet élément combine une exposition excessive des données et une affectation de masse.

JavaScript ✅ Filtrage des réponses 
// Renvoie uniquement les champs publics de l'objet utilisateur
function sanitizeUser(user, requesterId) {
  const publicFields = { id: user.id, name: user.name, avatar: user.avatar };

  // Seul le propriétaire peut voir les champs supplémentaires
  if (requesterId === user.id) {
    publicFields.email = user.email;
    publicFields.phone = user.phone;
  }

  return publicFields;
  // ❌ retourne l'utilisateur ; fuiterait password_hash etc.
}
  • Spécifiez explicitement les champs à inclure dans les réponses de l'API (approche de liste verte)
  • Restreindre les champs du corps de la demande acceptée à l'aide d'un schéma
  • Filtrer dynamiquement les champs de réponse en fonction des rôles

4️⃣ API4 : 2023 – Consommation de ressources sans restriction

High

Aperçu

Une vulnérabilité où il n'y a aucune limite sur la taille, la fréquence ou la consommation des ressources des requêtes API, conduisant à des attaques DoS ou à des augmentations de coûts.

JavaScript (Express) ✅ Mise en œuvre des limites de ressources 
const rateLimit = require('express-rate-limit');

// Limitation globale du débit
app.use(rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 100,                   // Maximum 100 requêtes
  standardHeaders: true,
}));

// Limite de taille de charge utile
app.use(express.json({ limit: '10kb' }));

// Limite supérieure de pagination
app.get('/api/items', (req, res) => {
  const limit = Math.min(parseInt(req.query.limit) || 20, 100);
  // ...
});

5️⃣ API5 : 2023 – Autorisation de niveau de fonction interrompue

High

Aperçu

Une vulnérabilité dans laquelle un contrôle d'accès approprié n'est pas mis en œuvre pour les fonctions d'administration et les points de terminaison, permettant aux utilisateurs réguliers d'effectuer des opérations administratives.

JavaScript ✅ Contrôle d'accès basé sur les rôles 
function authorize(...roles) {
  return (req, res, next) => {
    if (!roles.includes(req.user.role)) {
      return res.status(403).json({ error: 'Insufficient permissions' });
    }
    next();
  };
}

// Accès administrateur uniquement
app.delete('/api/admin/users/:id', authenticate, authorize('admin'), deleteUser);

// Administrateur et modérateur
app.put('/api/posts/:id/moderate', authenticate, authorize('admin', 'moderator'), moderatePost);

6️⃣ API6:2023 - Accès illimité aux flux commerciaux sensibles

Medium

Aperçu

Une vulnérabilité dans laquelle les flux logiques métier critiques ne sont pas protégés contre les attaques automatisées. Exemples : scalping de tickets, récolte massive de coupons, publication de spam, etc.

Atténuations

  • Implémenter CAPTCHA / reCAPTCHA
  • Empreinte digitale de l'appareil
  • Limitation des tarifs au niveau de l'entreprise sur les opérations
  • Détection des interactions humaines

7️⃣ API7 : 2023 – Forgerie de requêtes côté serveur (SSRF)

High

Aperçu

Une vulnérabilité où le serveur récupère une URL fournie par l'utilisateur sans validation, permettant un accès non autorisé aux réseaux internes.

JavaScript ✅ Atténuation SSRF 
const { URL } = require('url');

function isAllowedUrl(input) {
  try {
    const url = new URL(input);
    // Restreindre les protocoles
    if (!['https:', 'http:'].includes(url.protocol)) return false;
    // Bloquer les plages d'adresses IP privées
    const blocked = ['127.0.0.1', 'localhost', '0.0.0.0', '169.254.169.254'];
    if (blocked.includes(url.hostname)) return false;
    // Vérifiez les plages du réseau interne (10.x, 172.16-31.x, 192.168.x)
    if (/^(10\.|172\.(1[6-9]|2\d|3[01])\.|192\.168\.)/.test(url.hostname)) return false;
    return true;
  } catch {
    return false;
  }
}

8️⃣ API8:2023 - Mauvaise configuration de la sécurité

Medium

Aperçu

Paramètres de sécurité incorrects sur les serveurs ou frameworks API. Inclut l’autorisation de méthodes HTTP inutiles, les politiques CORS trop permissives, l’exposition des informations de débogage, etc.

JavaScript (Express) ✅ Configuration de l'en-tête de sécurité 
const helmet = require('helmet');
const cors = require('cors');

app.use(helmet());
app.use(cors({
  origin: ['https://app.example.com'], // N'utilisez pas de caractères génériques
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  credentials: true,
}));

// Gestionnaire d'erreurs : ne renvoie pas les traces de pile en production
app.use((err, req, res, next) => {
  res.status(500).json({
    error: process.env.NODE_ENV === 'production'
      ? 'Internal Server Error'
      : err.message,
  });
});

9️⃣ API9 : 2023 – Gestion inappropriée des stocks

Medium

Aperçu

Un état dans lequel les anciennes versions des points de terminaison de l'API restent exposées sans que les correctifs de sécurité soient appliqués. Également connues sous le nom d’API fantômes ou d’API zombies.

Atténuations

  • Documenter et gérer tous les points de terminaison de l'API (OpenAPI/Swagger)
  • Définir des calendriers de dépréciation clairs pour les API obsolètes
  • Bloquer les points de terminaison non enregistrés sur la passerelle API
  • Effectuer régulièrement des audits d'inventaire des API

🔟 API10:2023 - Consommation dangereuse des API

Medium

Aperçu

Une vulnérabilité où les réponses des API tierces sont fiables sans validation. Si une API externe est compromise, l'impact se propage à votre propre système.

JavaScript ✅ Validation des réponses API externes 
const Ajv = require('ajv');
const ajv = new Ajv();

// Valider les réponses de l'API externe avec un schéma
const externalSchema = {
  type: 'object',
  required: ['id', 'status'],
  properties: {
    id: { type: 'string', maxLength: 50 },
    status: { type: 'string', enum: ['active', 'inactive'] },
  },
  additionalProperties: false,
};

const validate = ajv.compile(externalSchema);

async function fetchExternalData() {
  const res = await fetch('https://api.external.com/data');
  const data = await res.json();

  if (!validate(data)) {
    throw new Error('External API response validation failed');
  }
  return data;
}
