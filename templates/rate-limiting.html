⏱ Rate Limiting

Rate limiting is a fundamental defense mechanism to protect your system from API abuse, DoS attacks, and scraping.

Rate Limiting Implementation in Express.js

JavaScript (Express)Multi-Layer Rate Limiting
const rateLimit = require('express-rate-limit');
const RedisStore = require('rate-limit-redis');

// Global limit: applied to all APIs
const globalLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 100,
  standardHeaders: true,    // Returns RateLimit-* headers
  legacyHeaders: false,
  message: { error: 'Too many requests, please try again later.' },
});

// For authentication endpoints: stricter limits
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,  // Login attempts limited to 5 per 15 minutes
  skipSuccessfulRequests: true, // Successful requests are not counted
});

// For distributed environments: Redis backend
const distributedLimiter = rateLimit({
  store: new RedisStore({ sendCommand: (...args) => redisClient.sendCommand(args) }),
  windowMs: 60 * 1000,
  max: 30,
});

app.use(globalLimiter);
app.use('/api/auth', authLimiter);

Rate Limiting Keys

By applying rate limits using composite keys such as API key, user ID, or endpoint in addition to IP address, you can minimize the impact on legitimate users.

🔍 Input Validation

All API inputs may contain malicious data. Always validate on the server side.

Validation Principles

Allowlist Approach

Define a list of permitted values. Safer than a denylist (blocklist). Can handle new attack patterns.

Type, Length & Range

Strictly check data types, maximum character counts, and numeric upper/lower bounds. A fundamental defense against buffer overflows.

Sanitize vs. Reject

Rejecting invalid input is safer than "fixing" it. Sanitization may produce unexpected transformations.

Validation with JSON Schema

JavaScript (Express + Ajv)Schema Validation
const Ajv = require('ajv');
const addFormats = require('ajv-formats');

const ajv = new Ajv({ allErrors: true, removeAdditional: true });
addFormats(ajv);

// Schema for user creation API
const createUserSchema = {
  type: 'object',
  required: ['name', 'email'],
  additionalProperties: false,
  properties: {
    name: {
      type: 'string',
      minLength: 1,
      maxLength: 100,
      pattern: '^[a-zA-Z0-9\\s\\-]+$', // Restrict to allowed characters
    },
    email: {
      type: 'string',
      format: 'email',
      maxLength: 254,
    },
    age: {
      type: 'integer',
      minimum: 0,
      maximum: 150,
    },
  },
};

// Validation middleware
function validateBody(schema) {
  const validate = ajv.compile(schema);
  return (req, res, next) => {
    if (!validate(req.body)) {
      return res.status(400).json({
        error: 'Validation failed',
        details: validate.errors,
      });
    }
    next();
  };
}

app.post('/api/users', validateBody(createUserSchema), createUser);

🌐 CORS (Cross-Origin Resource Sharing)

CORS is a mechanism that controls the browser's Same-Origin Policy. Misconfiguration can lead to serious security risks.

Dangerous Configuration

Access-Control-Allow-Origin: * and Access-Control-Allow-Credentials: true cannot be used together. Restrict wildcard usage to public APIs only.

Secure CORS Configuration

JavaScript (Express)CORS Configuration
const cors = require('cors');

// Explicitly specify allowed origins
const allowedOrigins = [
  'https://app.example.com',
  'https://admin.example.com',
];

app.use(cors({
  origin(origin, callback) {
    // Allow server-to-server communication (no origin)
    if (!origin || allowedOrigins.includes(origin)) {
      callback(null, true);
    } else {
      callback(new Error('Not allowed by CORS'));
    }
  },
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization', 'X-API-Key'],
  credentials: true,
  maxAge: 86400,  // Preflight cache: 24 hours
}));

CORS Checklist

  • Do not use a wildcard (*) for Access-Control-Allow-Origin (for APIs requiring authentication)
  • Access-Control-Allow-Methods should include only the required methods
  • Access-Control-Allow-Headers should include only the required headers
  • Cache preflight requests with Access-Control-Max-Age
  • Restrict exposed response headers with Access-Control-Expose-Headers
  • Never reflect the request Origin back directly in the response
